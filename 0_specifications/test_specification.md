# 测试验收规范

## 真实验收入口

- 本项目必须通过正式 Docker 部署入口进行真实业务验收，禁止用手动启动的临时服务替代正式入口。
- 真实验收时必须确认用户可以真实执行流程，不可以仅依靠代码、DOM 树或静态推断假设功能可用。
- 前端页面验收允许使用本机浏览器访问正式 Docker 暴露的前端入口。
- 当用户明确要求“和最终用户操作流程一致”时，业务主流程必须优先通过正式前端页面完成；API、数据库、日志只允许作为辅助取证，不得替代主流程验收。

## 测试环境选择
- 真实业务验收必须使用正式 Docker 入口，但这不等于所有自动化测试都必须在临时 Maven Docker 容器中执行。
- 后端自动化测试必须优先使用本机环境， 之后再考虑 `12_test_env/test_env.md` 中声明的本机 JDK/Maven 工具链，并确认 Docker Desktop 正常运行。

## 自动化测试结果判定

- 验收报告必须记录自动化测试的执行命令、执行环境、测试总数、失败数、错误数和跳过数。
- 不能只依据 `0 failures / 0 errors` 判定后端测试通过，必须同时检查 `skipped`。
- 如果相关测试被 skipped，本轮数据库集成测试只能标记为“未完整执行”或“测试环境限制”，不得等同于完整通过。
- 若仍需在 Maven Docker 容器中执行后端测试，必须先验证容器内可以访问 Docker，并在报告中说明验证方式。

## 敏感信息

- 真实 LLM 调用可使用 `12_test_env/test_env.md` 中的测试环境信息，但 API Key 不得写入验收报告、错误记录或对外说明。
