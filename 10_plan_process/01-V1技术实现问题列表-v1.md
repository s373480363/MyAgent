# V1 技术实现口径清单 v1

## 1. 文档目的

本文档用于冻结完整 V1 开发过程中的关键技术实现口径，作为开发、测试和联调统一执行依据。

当前 Q01 至 Q08 已全部得到明确结论，本版不再存在待确认项。

## 2. 状态说明

| 状态 | 说明 |
|------|------|
| 已确认 | 已经得到明确结论，可直接进入实现 |
| 已关闭 | 已写入正式设计或实现约定，不再作为问题跟踪 |

## 3. 优先级说明

| 优先级 | 含义 |
|--------|------|
| P0 | 不确认会直接影响正确性或主链路实现 |
| P1 | 会影响实现方式、测试口径或错误处理，但可在约定后推进 |
| P2 | 影响优化方式或工程细节，不影响主链路正确性 |

## 4. 已冻结的技术实现口径

| 编号 | 优先级 | 状态 | 技术主题 | 架构结论 | 影响范围 |
|------|--------|------|----------|----------------|----------|----------|
| Q01 | P0 | 已确认 | JSONPath 映射写回语义 | 只支持受控 JSONPath 子集：`$`、点路径、数组下标。读取路径不存在、数组越界、类型不匹配直接失败；写回禁止覆盖 `$.input`，缺失中间对象自动创建，不自动创建数组，父路径类型冲突直接失败，目标字段允许显式覆盖，`null` 允许写回并交由 Schema 校验决定是否通过。 | `MappingService`、工作流发布校验、节点执行器、前端映射编辑器和测试用例 |
| Q02 | P0 | 已确认 | 运行终止和状态传播 | `FAILED` 只表示业务失败；`TIMEOUT` 表示运行或节点自身超时；`CANCELED` 表示外部中断或父运行级联取消。父运行超时或取消时，必须尽力级联取消活跃子运行和外部进程；`AGENT_CALL` 子运行超时则父节点与父运行按 `TIMEOUT` 处理。 | 运行内核、Trace、运行详情和异常状态展示 |
| Q03 | P1 | 已确认 | Schema 草稿保存语义 | Schema 草稿采用同一条 `DRAFT` 记录原地更新；只有显式创建新版本或旧版本已锁定需演进时才生成新的版本行。 | Schema 页面、版本追溯、锁定规则和数据库更新方式 |
| Q04 | P1 | 已确认 | LLM 结构化输出接入方式 | 配置 `outputSchemaRef` 时统一走结构化输出模式；未配置时走普通文本输出模式。必须区分模型调用失败、结构化响应解析失败、Schema 校验失败三类错误。 | `model` 模块、LLM 节点、错误码和 Trace 事件内容 |
| Q05 | P2 | 已确认 | Trace 全文保存策略 | V1 默认保存全文。`NodeRun.inputJson/outputJson`、`MODEL_REQUEST`、`MODEL_RESPONSE`、工具/方法结构化输入输出保存全文；外部 Agent 的 `stdout/stderr/Git diff` 仅在采集开关开启时保存全文。列表页只读取 `summary`，详情页读取 `detailJson`。 | `trace_event`、`node_run`、数据库体积和详情页性能 |
| Q06 | P1 | 已确认 | Java 方法参数转换失败处理 | JSON->POJO、POJO->JSON、序列化失败统一归为 `JAVA_METHOD` 执行失败；只有输入不符合 `inputSchema` 或输出不符合 `outputSchema` 时才归为 Schema 校验失败。 | `JavaMethodNodeExecutor`、错误码、Trace 事件和测试断言 |
| Q07 | P1 | 已确认 | 外部 Agent 结果解析规则 | 统一使用 `ExternalAgentResult` 作为标准结果模型。优先按 `resultSource` 提取 `outputJson`；无法提取时退化为统一文本摘要。`stdout/stderr` 是调试字段，不自动等于业务输出。 | `ExternalAgentAdapter`、节点执行器和运行详情页展示 |
| Q08 | P2 | 已确认 | Eval 评分和断言执行顺序 | 执行顺序固定为“outputSchema 校验 -> 确定性断言 -> 可选 LLM 评分”。LLM 评分只做辅助，不覆盖正式断言；关键用例失败必须影响整次验收结论。 | `EvalExecutor`、验收结果解释和页面展示 |

## 5. 建议落地顺序

建议按下面顺序落地：

1. 先按 `Q01` 和 `Q02` 的已确认规则实现 `MappingService`、运行状态机和状态传播。
2. 再按 `Q03`、`Q04`、`Q06`、`Q07` 的已确认规则实现 Schema、LLM、JAVA_METHOD、EXTERNAL_AGENT。
3. 最后按 `Q05`、`Q08` 的已确认规则收口 Trace 存储和 Eval 执行。

## 6. 当前执行建议

当前问题已由架构侧确认，可按以下规则直接进入实现：

- JSONPath 映射严格按受控子集和统一失败规则落地。
- 运行状态映射采用单一规则表，不允许节点自行解释状态。
- Schema 草稿、LLM 输出、外部 Agent 结果按统一结果模型和统一错误分类处理。
- Trace 默认按全文保存，后续如需控制体积，再追加压缩或清理机制。
