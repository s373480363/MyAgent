# Agent 管理平台 - 01 产品定位与交付边界 v1

## 1. 产品定位

Agent 管理平台是一套多 Agent 管理与编排平台。平台允许用户创建 Agent 实例，为 Agent 配置内部工作流，并通过平台页面或后端接口同步调用 Agent。

平台的核心不是“写一个固定 Agent”，而是提供一套生成、管理、执行、调试和追踪 Agent 的机制。Agent 的行为来自工作流、节点配置、模型参数、Java 方法、外部 Agent、工具和结构化数据契约。

## 2. 产品原则

| 原则 | 说明 |
|---|---|
| 可配置优先 | Agent 行为尽量来自工作流和节点配置，不硬编码在业务代码中。 |
| 能力节点化 | LLM、Java 方法、外部 Agent、工具、条件、审核、总结、Agent 调用都表现为节点。 |
| Schema 驱动 | 节点输入输出、Java POJO、前端表单、模型结构化输出围绕同一套 JSON Schema。 |
| 执行可追踪 | 每次运行必须能看到节点输入、节点输出、耗时、错误和 Trace。 |
| 用户主导验收 | 提示词验收集由用户定义或确认，AI 只能辅助起草和解释。 |
| 平台统一调度 | 内部 Agent 互调必须通过平台创建运行记录和追踪链路。 |
| 外部 Agent 轻量接入 | 平台只负责统一调用、结果写回、错误可见和必要调试记录。 |

## 3. 部署边界

| 项目 | V1 规则 |
|---|---|
| 部署环境 | 本机或内网部署。 |
| 用户模型 | 单用户模型。 |
| 多人访问 | 允许多人访问同一个服务，但属于信任环境内共享。 |
| 用户隔离 | 不提供用户隔离。 |
| 权限控制 | 不提供登录、角色、权限、多租户。 |
| 公网暴露 | 不按公网 SaaS 安全边界设计，不建议直接暴露到公网。 |

## 4. OpenAI 接入边界

| 项目 | V1 规则 |
|---|---|
| 模型供应商 | 只支持 OpenAI 官方 API。 |
| Azure OpenAI | 不进入 V1。 |
| 默认模型 | 不在产品设计中写死，由后端配置项管理，例如 `myagent.openai.default-model`。 |
| 节点模型覆盖 | LLM 类节点可以覆盖默认模型。 |
| 结构化输出 | 节点配置 `outputSchema` 时强制校验；未配置时按普通文本输出处理。 |

## 5. V1 开发范围边界

V1 不是只交付最小聊天流程，而是交付完整的第一版 Agent 管理平台能力。以下内容均属于 V1 需要开发的产品能力：

- Agent 管理
- 工作流画布
- 工作流版本管理
- LLM 节点
- Java 方法节点
- 外部 Agent 节点
- 工具节点
- 条件节点
- 审核节点
- 总结节点
- 内部 Agent 调用节点
- Schema 管理
- 节点验收
- 同步运行
- Trace 追踪
- 运行记录

## 6. V1 不包含内容

- 登录、权限、角色、多租户。
- Azure OpenAI。
- 复杂 RAG 知识库。
- 长期记忆系统。
- 异步队列。
- 人工审批。
- 完整工作流验收。
- 插件市场。
- 自动优化提示词。
- 外部 Agent 深度托管。

## 7. 默认运行限制

默认限制用于保护同步工作流，不代表业务规则不可变。实现时应允许在 Agent 或节点层覆盖。

| 限制项 | 建议默认值 |
|---|---:|
| 单次 Agent 运行超时 | 10 分钟 |
| 单个 LLM 节点超时 | 2 分钟 |
| 单个 JAVA_METHOD 节点超时 | 30 秒 |
| 单个 EXTERNAL_AGENT 节点超时 | 10 分钟 |
| 最大工作流步数 | 30 |
| 最大 Agent 调用深度 | 3 |

## 8. 版本不可变原则

- 每个 Agent 只能有一个当前草稿。
- 每次发布生成一个不可变工作流版本。
- 正式 API 调用默认使用当前发布版本。
- 草稿可以调试运行，但不能作为正式 API 调用版本。
- 运行记录必须绑定具体 `workflowVersionId`。
- 已发布工作流引用的 Schema 版本必须只读。
- Schema 变更通过新版本演进。
