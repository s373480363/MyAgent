# 10-节点验收架构设计 v1

## 1. 定位

节点验收用于验证单个 LLM 类节点的提示词、模型参数、结构化输出和自然语言能力。v1 不做完整工作流验收。

支持节点：

- LLM。
- REVIEW。
- SUMMARY。

LLM 类节点验收的主判断方式是自然语言 `judgeRule` + judge LLM 结构化判定。确定性检查只作为可选硬约束，不再主导正式通过率。

## 2. 领域对象

| 对象 | 职责 |
|------|------|
| EvalSuite | 绑定 Agent、WorkflowVersion、节点，保存验收目标、judge 模型供应项和通过阈值 |
| EvalCase | 保存输入、参考样例、自然语言 judgeRule、可选 hardChecks、关键用例标记、确认状态和来源追踪 |
| EvalRun | 保存一次验收运行汇总结果、历史对比数据和运行关联信息 |
| EvalCaseResult | 保存单条用例执行输出、hardCheckResults、judgeResult 和通过状态 |
| EvalExecutor | 执行目标节点、硬约束和 judge LLM，产出结果 |

## 3. EvalSuite

关键字段：

- suiteId。
- agentId。
- workflowVersionId。
- nodeId。
- name。
- goal。
- judgeModelOfferingKey。
- judgeTemperature。
- passThreshold。
- status：DRAFT、CONFIRMED、ARCHIVED。

状态流转：

- `DRAFT`：可编辑，允许创建和修改用例，尚未进入正式验收。
- `CONFIRMED`：已确认，可执行正式验收运行。
- `ARCHIVED`：已归档，不再允许新的正式验收运行。

规则：

- `judgeModelOfferingKey` 必须引用已启用模型供应项。
- 同一 EvalSuite 内的 EvalCase 默认使用同一个 judge 模型供应项。
- EvalSuite 确认前必须至少存在一个可运行 EvalCase。

## 4. EvalCase

关键字段：

- caseId。
- suiteId。
- inputJson。
- referenceSampleJson。
- judgeRuleText。
- hardChecksJson。
- critical。
- confirmStatus：USER_CREATED、USER_CONFIRMED、AI_DRAFT_PENDING、ARCHIVED。
- sourceAgentRunDbId。
- sourceRunId。
- sourceNodeRunId。
- sourceWorkflowVersionId。
- sourceNodeId。

其中：

- `referenceSampleJson` 是参考样例，不是标准答案。
- `judgeRuleText` 是主验收规则，用户确认 EvalCase 前必须填写。
- `hardChecksJson` 是可选硬约束，用于结构、字段、枚举、范围、正则等检查。
- `sourceAgentRunDbId` 对应数据库 `agent_run.id`。
- `sourceRunId` 对应数据库 `agent_run.run_no`，仅用于对外追溯和页面展示。
- `sourceNodeRunId` 对应数据库 `node_run.id`。

用户手工创建用例默认是 USER_CREATED，AI 生成用例默认是 AI_DRAFT_PENDING，二者都不能计入正式通过率。只有 USER_CONFIRMED 可以进入正式 EvalRun。

## 5. Judge Rule

`judgeRule` 是用户用自然语言定义的验收规则。它是 LLM 节点验收的主判断依据。

judge LLM 输入必须包含：

- EvalCase 输入。
- 目标节点实际输出。
- 参考样例。
- judgeRule。
- hardCheckResults。
- 目标节点类型和节点标识。

judge LLM 必须输出结构化 JSON：

```json
{
  "passed": true,
  "score": 86,
  "reason": "回答覆盖了关键风险点，未发现编造内容。",
  "criteriaResults": [
    {
      "criterion": "覆盖关键风险点",
      "passed": true,
      "score": 30,
      "reason": "三个风险点均已覆盖。"
    }
  ]
}
```

## 6. Hard Checks

hardChecks 是可选硬约束。

V1 支持类型和必填字段：

- `SCHEMA_VALIDATION`：只需要 `type`，目标节点必须配置 outputSchema。
- `JSON_PATH_EXISTS`：需要 `type` 和 `path`。
- `JSON_PATH_IN`：需要 `type`、`path` 和非空数组 `values`。
- `JSON_PATH_NUMBER_RANGE`：需要 `type`、`path`，并且 `min` 和 `max` 至少一个存在。
- `JSON_PATH_REGEX`：需要 `type`、`path` 和合法 Java 正则 `pattern`。
- `JSON_PATH_CONTAINS`：需要 `type`、`path` 和非空字符串 `expected`。
- `JSON_PATH_NOT_CONTAINS`：需要 `type`、`path` 和非空字符串 `expected`。

hardChecks 不用于自然语言主质量判断。长文本摘要、解释、开放式回答不得依赖精确相等作为主要验收方式。V1 不提供 `JSON_PATH_EQUALS`；需要稳定单值判断时使用 `JSON_PATH_IN` 且只传入一个允许值。

hardCheckResults 必须是数组，未配置 hardChecks 时为空数组。每项至少包含 `type`、`passed`、`message`，可包含 `path`、`expected`、`actual` 和 `details`。

## 7. EvalSuite 与 EvalCase 流程

- EvalSuite 需要支持创建、更新、确认、归档、查询和运行。
- EvalCase 需要支持创建、更新、确认、归档和查询。
- EvalRun 需要支持列表、详情、结果明细和历史对比查询。
- 只有 `EvalSuite.status=CONFIRMED` 且 `EvalCase.confirmStatus=USER_CONFIRMED` 的用例才允许进入正式验收运行。
- EvalCase 确认前必须存在有效 judgeRule。

## 8. 执行流程

```text
选择 EvalSuite
  -> 读取绑定 WorkflowVersion 和节点定义
  -> 过滤已确认 EvalCase
  -> 逐条构造节点输入
  -> 执行目标 LLM 类节点
  -> 执行 hardChecks
  -> hardChecks 失败则写失败结果并跳过 judge LLM
  -> hardChecks 通过则调用 judge LLM
  -> 解析 judgeResult
  -> 创建 AgentRun(runType=EVAL)，用于统一 Trace 时间线
  -> 写 EvalRun、EvalCaseResult 和历史对比统计
  -> 写 EVAL_CASE_RESULT TraceEvent
```

## 9. 通过规则

- 普通用例按 EvalCaseResult.passed 统计通过率。
- hardChecks 失败时，用例直接失败。
- hardChecks 失败时不调用 judge LLM，`judgeResult`、`judgeRawText`、`judgeModelOfferingKey` 和 `judgePromptVersion` 都为空。
- judge 调用失败、输出非 JSON 或缺少 `passed` 字段时，用例失败。
- 关键用例失败时，即使整体通过率达标，也应提示验收失败。
- 未确认用例不计入正式通过率。
- judge LLM 必须通过模型供应项路由调用，与 LLM、REVIEW、SUMMARY 节点共用模型网关和 Trace 安全 DTO。

## 10. 数据关联建议

节点验收必须以 EvalRun 和 EvalCaseResult 作为正式结果来源。

为了统一 Trace 展示，v1 验收运行必须创建 `AgentRun(runType=EVAL)`。该 AgentRun 只承载运行上下文和 Trace 时间线，不替代 EvalRun。`EvalRun.agentRunId` 必须稳定关联该运行记录，TraceEvent 必须同时写入 `runId` 和 `evalRunId`，避免新增未定义的 EvalTraceEvent 模型，也避免验收 Trace 与普通运行 Trace 分叉。

EvalCaseResult 必须保存：

- 目标节点实际输出。
- hardCheckResults。
- judgeResult。
- judge 原始文本。
- judge 模型供应项。
- judge 提示词版本。
- 是否通过。
- errorMessage。

分数只存在于 `judgeResult.score`，不设置 EvalCaseResult 顶层分数字段。

## 11. 从 NodeRun 创建 EvalCase

平台必须支持从历史 NodeRun 生成待确认 EvalCase。

能力要求：

- 复制节点输入作为用例输入。
- 复制节点输出作为参考样例。
- 默认生成 `AI_DRAFT_PENDING` 用例。
- 记录来源链路，至少包含 `sourceAgentRunDbId`、`sourceRunId`、`sourceNodeRunId`、`sourceWorkflowVersionId` 和 `sourceNodeId`。
- 生成后的用例仍需由用户补充 judgeRule 并确认后，才能进入正式通过率统计。

## 12. EvalRun 查询能力

EvalRun 作为一等对象，必须支持：

- 列表查询。
- 详情查询。
- 按 `suiteId` 查询历史运行。
- 查询失败用例明细。
- 查询通过率历史对比数据。

对外展示时，EvalRun 结果应包含 `passRate`、`totalCaseCount`、`passedCaseCount`、`failedCaseCount` 和 `summary`。
`GET /api/eval-runs/{evalRunId}` 使用独立详情 DTO 承载验收汇总、失败摘要、judge 摘要和历史对比；`GET /api/eval-runs/{evalRunId}/results` 使用分页结果明细 DTO 承载逐条用例执行细节，不再由开发侧自行拼装字段。
