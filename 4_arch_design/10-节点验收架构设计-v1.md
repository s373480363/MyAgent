# 10-节点验收架构设计 v1

## 1. 定位

节点验收用于验证单个 LLM 类节点的提示词、模型参数和结构化输出。v1 不做完整工作流验收。

支持节点：

- LLM。
- REVIEW。
- SUMMARY。

## 2. 领域对象

| 对象 | 职责 |
|------|------|
| EvalSuite | 绑定 Agent、WorkflowVersion、节点，保存验收目标和通过阈值 |
| EvalCase | 保存输入、参考答案、断言规则、评分规则、关键用例标记、确认状态和来源追踪 |
| EvalRun | 保存一次验收运行汇总结果、历史对比数据和运行关联信息 |
| EvalCaseResult | 保存单条用例执行结果 |
| EvalExecutor | 执行验收用例，产出结果 |

## 3. EvalSuite

关键字段：

- suiteId。
- agentId。
- workflowVersionId。
- nodeId。
- name。
- goal。
- passThreshold。
- status：DRAFT、CONFIRMED、ARCHIVED。

状态流转：

- `DRAFT`：可编辑，允许创建和修改用例，尚未进入正式验收。
- `CONFIRMED`：已确认，可执行正式验收运行。
- `ARCHIVED`：已归档，不再允许新的正式验收运行。

## 4. EvalCase

关键字段：

- caseId。
- suiteId。
- inputJson。
- referenceAnswerJson。
- assertionsJson。
- scoreRuleJson。
- critical。
- confirmStatus：USER_CREATED、USER_CONFIRMED、AI_DRAFT_PENDING、ARCHIVED。
- sourceAgentRunDbId。
- sourceRunId。
- sourceNodeRunId。
- sourceWorkflowVersionId。
- sourceNodeId。

其中：

- `sourceAgentRunDbId` 对应数据库 `agent_run.id`。
- `sourceRunId` 对应数据库 `agent_run.run_no`，仅用于对外追溯和页面展示。
- `sourceNodeRunId` 对应数据库 `node_run.id`。

AI 生成用例默认是 AI_DRAFT_PENDING，不能计入正式通过率。

## 5. EvalSuite 与 EvalCase 流程

- EvalSuite 需要支持创建、更新、确认、归档、查询和运行。
- EvalCase 需要支持创建、更新、确认、归档和查询。
- EvalRun 需要支持列表、详情、结果明细和历史对比查询。
- 只有 `EvalSuite.status=CONFIRMED` 且已确认的 EvalCase 才能进入正式验收运行。

## 6. 断言类型

- Schema 校验。
- 字段等于。
- 包含 / 不包含。
- 正则匹配。
- 数值范围。
- 枚举校验。
- JSONPath 断言。
- 可选 LLM 评分。

## 7. 执行流程

```text
选择 EvalSuite
  -> 读取绑定 WorkflowVersion 和节点定义
  -> 过滤已确认 EvalCase
  -> 逐条构造节点输入
  -> 执行目标 LLM 类节点
  -> 校验 outputSchema
  -> 执行确定性断言
  -> 可选执行 LLM 评分
  -> 创建 AgentRun(runType=EVAL)，用于统一 Trace 时间线
  -> 写 EvalRun、EvalCaseResult 和历史对比统计
  -> 可选写 EVAL_CASE_RESULT TraceEvent
```

## 8. 通过规则

- 普通用例按通过率计算。
- 关键用例失败时，即使整体通过率达标，也应提示验收失败。
- 未确认用例不计入正式通过率。
- LLM 评分是可选辅助，不替代用户定义的正式断言。

## 9. 数据关联建议

节点验收必须以 EvalRun 和 EvalCaseResult 作为正式结果来源。

为了统一 Trace 展示，v1 验收运行必须创建 `AgentRun(runType=EVAL)`。该 AgentRun 只承载运行上下文和 Trace 时间线，不替代 EvalRun。`EvalRun.agentRunId` 必须稳定关联该运行记录，TraceEvent 必须同时写入 `runId` 和 `evalRunId`，避免新增未定义的 EvalTraceEvent 模型，也避免验收 Trace 与普通运行 Trace 分叉。

## 10. 从 NodeRun 创建 EvalCase

平台必须支持从历史 NodeRun 生成待确认 EvalCase。

能力要求：

- 复制节点输入作为用例输入。
- 复制节点输出作为参考答案。
- 默认生成 `AI_DRAFT_PENDING` 用例。
- 记录来源链路，至少包含 `sourceAgentRunDbId`、`sourceRunId`、`sourceNodeRunId`、`sourceWorkflowVersionId` 和 `sourceNodeId`。
- 生成后的用例仍需由用户补充断言规则并确认后，才能进入正式通过率统计。

## 11. EvalRun 查询能力

EvalRun 作为一等对象，必须支持：

- 列表查询。
- 详情查询。
- 按 `suiteId` 查询历史运行。
- 查询失败用例明细。
- 查询通过率历史对比数据。

对外展示时，EvalRun 结果应包含 `passRate`、`totalCaseCount`、`passedCaseCount`、`failedCaseCount` 和 `summary`。
`GET /api/eval-runs/{evalRunId}` 使用独立详情 DTO 承载验收汇总、失败摘要和历史对比；`GET /api/eval-runs/{evalRunId}/results` 使用分页结果明细 DTO 承载逐条用例执行细节，不再由开发侧自行拼装字段。
