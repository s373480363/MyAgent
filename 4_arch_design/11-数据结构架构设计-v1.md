# 11-数据结构架构设计 v1

## 1. 存储原则

- PostgreSQL 作为唯一主存储。
- 关系字段保存标识、状态、版本、时间、外键和筛选字段。
- JSONB 保存工作流节点、边、节点配置、运行输入输出、Trace payload 和 JSON Schema。
- 可执行工作流定义不可变；正式运行和调试运行都必须绑定具体 `workflow_version.id`。
- 发布版本不可变。
- 被发布工作流引用的 Schema 版本只读。

## 2. 核心表

| 表 | 说明 |
|----|------|
| agent_definition | Agent 基础信息、启停状态、草稿版本指针和当前发布版本指针 |
| workflow_version | 不可变工作流定义记录，Agent 通过指针引用当前草稿、当前发布版本和历史版本 |
| schema_definition | JSON Schema 契约和版本 |
| java_method_definition | 已注册 Java 方法目录 |
| tool_definition | 已注册工具目录 |
| external_agent_definition | 外部 Agent 适配器目录 |
| agent_run | 一次 Agent 运行记录 |
| node_run | 一次节点执行记录 |
| trace_event | 运行时间线事件，包含摘要和详情 |
| agent_message | Agent 调用 Agent 的父子运行关系 |
| eval_suite | 节点验收集 |
| eval_case | 节点验收用例 |
| eval_run | 节点验收运行汇总 |
| eval_case_result | 节点验收用例结果 |
| system_setting | 白名单系统配置覆盖值，如默认模型与运行限制 |

## 3. 关键字段

### 3.1 agent_definition

- description。

- system_prompt。
- default_model。
- temperature。
- timeout_seconds。
- max_steps。
- current_draft_workflow_version_id。
- current_published_workflow_version_id。
- created_at。
- updated_at。

### 3.2 workflow_version

- id。
- agent_id。
- version_no，工作流定义序号，从 1 开始递增。
- status：DRAFT、PUBLISHED、HISTORY。
- nodes_json。
- edges_json。
- runtime_options_json，正式结构固定为 `{ timeoutSeconds, maxSteps, maxAgentCallDepth }`。
- referenced_schema_versions_json，正式结构固定为对象数组，每项为 `{ schemaId, schemaKey, version }`。
- source_workflow_version_id，可空，用于记录从哪个版本复制而来。
- published_at。
- created_at。
- updated_at。

补充约束：

- `workflow_version` 是运行事实来源。任何被 AgentRun 绑定过的工作流定义都必须能原样还原。
- 每个 Agent 最多一个当前 `DRAFT`，由 `agent_definition.current_draft_workflow_version_id` 指向。
- 每个 Agent 最多一个当前 `PUBLISHED`，由 `agent_definition.current_published_workflow_version_id` 指向。
- 保存草稿时不允许原地覆盖当前 `DRAFT` 行；必须创建新的 `DRAFT` 行，写入新的 `version_no`，并更新当前草稿指针。
- 被替换的旧 `DRAFT` 行转为 `HISTORY`，内容保持不变；如果历史调试运行绑定了它，仍可通过 `workflow_version_id` 完整复盘。
- 发布时不允许把当前 `DRAFT` 行原地转换为 `PUBLISHED`；必须复制当前 `DRAFT` 内容创建新的 `PUBLISHED` 行。
- 发布成功后更新当前发布指针；当前草稿指针可以继续指向本次发布来源草稿，直到用户再次保存草稿并生成新的 `DRAFT` 版本。
- 旧 `PUBLISHED` 只允许状态转为 `HISTORY`，不得修改节点、边、映射、运行配置和 Schema 引用。
- `HISTORY` 表示不再作为当前草稿或当前发布使用的不可变版本，可能来自旧草稿快照或旧发布版本。
- 调试运行和正式运行都必须绑定 `workflow_version.id`；调试草稿时绑定当前不可变 `DRAFT`，后续保存草稿生成新 `DRAFT` 不影响旧运行。
- `runtime_options_json.timeoutSeconds`、`runtime_options_json.maxSteps` 和 `runtime_options_json.maxAgentCallDepth` 是已持久化版本的运行时事实来源；执行时不得再回头读取 Agent 当前主数据重算。
- `agent_definition.timeout_seconds` 和 `agent_definition.max_steps` 只作为创建或归一化 WorkflowVersion 时的默认值来源，不是已持久化版本的最终运行真相。
- `referenced_schema_versions_json` 由后端根据节点中的直接 Schema 引用派生、去重并稳定排序；不允许同时支持字符串列表和对象列表两种结构。

### 3.11 system_setting

`system_setting` 采用 key-value 模式，因为 v1 配置项数量少、变化频率低，并且不同配置项类型差异较大。关系型强字段配置表会导致每次新增配置都需要修改表结构，不符合 v1 简洁目标。

- setting_key，主键。
- setting_value。
- value_type：STRING、NUMBER、BOOLEAN、JSON。
- description。
- editable。
- updated_at。

v1 只允许以下 `setting_key` 进入该表：

- `myagent.openai.default-model`
- `myagent.runtime.default-agent-timeout-seconds`
- `myagent.runtime.default-llm-timeout-seconds`
- `myagent.runtime.default-java-method-timeout-seconds`
- `myagent.runtime.default-external-agent-timeout-seconds`
- `myagent.runtime.default-max-steps`
- `myagent.runtime.default-max-agent-call-depth`

补充约束：

- `system_setting` 只保存白名单配置的覆盖值，不镜像全部启动配置。
- `myagent.openai.api-key` 和 `myagent.trace.persist-full-model-content` 不进入此表。
- `myagent.runtime.default-timeout-seconds` 不是 v1 合法 `setting_key`。

## 4. 枚举

| 枚举 | 取值 |
|------|------|
| Agent 状态 | ENABLED、DISABLED |
| WorkflowVersion 状态 | DRAFT、PUBLISHED、HISTORY |
| 运行类型 | DEBUG、API、AGENT_CALL、EVAL |
| 运行状态 | PENDING、RUNNING、SUCCESS、FAILED、TIMEOUT、CANCELED |
| 节点类型 | START、LLM、CONDITION、JAVA_METHOD、AGENT_CALL、EXTERNAL_AGENT、TOOL、REVIEW、SUMMARY、END |
| Schema 来源 created_from | USER_CREATED、JAVA_METHOD_SCAN、TOOL_DEFINITION、SYSTEM_BUILTIN、AGENT_INPUT、AGENT_OUTPUT |
| Schema 状态 | DRAFT、ACTIVE、ARCHIVED |
| EvalCase 确认状态 | USER_CREATED、USER_CONFIRMED、AI_DRAFT_PENDING、ARCHIVED |

## 5. 索引方向

详细索引在数据库设计中展开，架构层建议至少考虑：

- agent_definition.agent_key 唯一索引。
- workflow_version(agent_id, status, version_no)。
- workflow_version(source_workflow_version_id)。
- schema_definition(schema_key, version) 唯一索引。
- schema_definition(created_from, status)。
- agent_run(agent_id, status, started_at)。
- agent_run(agent_key, run_type, started_at)。
- node_run(run_id, node_id)。
- trace_event(run_id, event_time)。
- trace_event(eval_run_id, event_time)。
- eval_suite(agent_id, workflow_version_id, node_id)。
- eval_case(suite_id, confirm_status)。
- eval_run(suite_id, status)。
