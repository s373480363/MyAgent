# 12-公共接口架构设计 v1

## 1. 接口原则

- REST API 统一前缀为 `/api`。
- v1 不做登录鉴权。
- 所有错误消息使用中文。
- 成功和失败响应结构稳定。
- 同步运行失败也必须返回 runId。
- 详细字段在 `7_interface_design` 中定义。
- 后端必须输出 OpenAPI 契约，前端 TypeScript 类型由 OpenAPI 生成。
- `GET /api/runs/{runId}`、`GET /api/eval-runs/{evalRunId}`、`GET /api/eval-runs/{evalRunId}/results` 和 `GET /api/eval-suites/{suiteId}/run-history` 在 v1 中必须冻结为正式响应 DTO，不保留“建议字段由开发补齐”的口径。

## 2. 响应规范

成功响应：

```json
{
  "success": true,
  "data": {},
  "error": null
}
```

失败响应：

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "WORKFLOW_VALIDATION_FAILED",
    "message": "工作流校验失败，请修正标记节点后重新发布。",
    "details": []
  }
}
```

## 3. API 分组

| 分组 | 接口方向 | 说明 |
|------|----------|------|
| Agent | `GET /api/agents`、`POST /api/agents`、`PUT /api/agents/{agentId}`、`PUT /api/agents/{agentId}/status`、`GET /api/agents/{agentId}` | Agent 管理和启用停用；详情接口返回当前草稿、当前发布和历史版本入口摘要 |
| 工作流草稿 | `GET /api/agents/{agentId}/workflow-draft`、`PUT /api/agents/{agentId}/workflow-draft`、`POST /api/agents/{agentId}/workflow-draft/copy-from-version` | 每个 Agent 一个当前草稿；保存草稿生成新的不可变 DRAFT 版本并更新指针；复制历史版本必须走显式命令接口 |
| 工作流发布 | `POST /api/agents/{agentId}/workflow-draft/validate`、`POST /api/agents/{agentId}/workflow-draft/publish` | 校验并发布不可变版本 |
| 工作流版本 | `GET /api/agents/{agentId}/workflow-versions`、`GET /api/agents/{agentId}/workflow-versions/{workflowVersionId}` | 查看当前草稿、当前发布版本和历史版本；历史版本只读 |
| 同步运行 | `POST /api/agents/{agentKey}/runs` | 正式 API 调用当前发布版本 |
| 调试运行 | `POST /api/agents/{agentId}/debug-runs` | 可运行草稿或发布版本 |
| 运行查询 | `GET /api/runs`、`GET /api/runs/{runId}` | 查询 AgentRun、NodeRun、TraceEvent |
| Schema | `GET /api/schemas`、`POST /api/schemas`、`PUT /api/schemas/{schemaId}`、`GET /api/schemas/{schemaId}`、`POST /api/schemas/{schemaId}/versions` | Schema 管理 |
| 模型供应商 | `GET /api/model-providers`、`POST /api/model-providers`、`GET /api/model-providers/{providerId}`、`PUT /api/model-providers/{providerId}`、`PUT /api/model-providers/{providerId}/secrets`、`PUT /api/model-providers/{providerId}/status`、`POST /api/model-providers/{providerId}/test` | 模型供应商管理、密钥维护和连接测试 |
| 模型供应项 | `GET /api/model-offerings`、`GET /api/model-offerings/{offeringKey}`、`GET /api/model-offerings/by-keys`、`POST /api/model-offerings`、`PUT /api/model-offerings/{offeringId}`、`PUT /api/model-offerings/{offeringId}/status` | 供应商下可调用模型入口管理和已绑定值回填 |
| Java 方法 | `GET /api/java-methods`、`GET /api/java-methods/{methodId}` | 查看已注册方法 |
| 工具 | `GET /api/tools`、`GET /api/tools/{toolId}` | 查看已注册工具 |
| 外部 Agent | `GET /api/external-agents`、`GET /api/external-agents/{adapterId}`、`POST /api/external-agents`、`PUT /api/external-agents/{adapterId}`、`PUT /api/external-agents/{adapterId}/secrets`、`PUT /api/external-agents/{adapterId}/status`、`POST /api/external-agents/{adapterId}/test` | 查看、维护、单独更新敏感 header secret 和测试外部 Agent 适配器 |
| 节点验收 | `GET /api/eval-suites`、`POST /api/eval-suites`、`PUT /api/eval-suites/{suiteId}`、`PUT /api/eval-suites/{suiteId}/confirm`、`PUT /api/eval-suites/{suiteId}/archive`、`GET /api/eval-suites/{suiteId}/cases`、`POST /api/eval-suites/{suiteId}/cases`、`GET /api/eval-suites/{suiteId}/cases/{caseId}`、`PUT /api/eval-suites/{suiteId}/cases/{caseId}`、`PUT /api/eval-suites/{suiteId}/cases/{caseId}/confirm`、`PUT /api/eval-suites/{suiteId}/cases/{caseId}/archive`、`POST /api/node-runs/{nodeRunId}/eval-cases`、`POST /api/eval-suites/{suiteId}/runs`、`GET /api/eval-suites/{suiteId}/runs`、`GET /api/eval-runs/{evalRunId}`、`GET /api/eval-runs/{evalRunId}/results`、`GET /api/eval-suites/{suiteId}/run-history` | 管理验收、查询历史运行、查看失败明细、历史对比和从 NodeRun 生成用例 |
| 系统设置 | `GET /api/settings`、`PUT /api/settings` | 仅暴露 v1 白名单中的非密钥运行限制配置 |

补充原则：

- `GET /api/settings` 和 `PUT /api/settings` 只处理白名单中的运行限制设置键，不暴露模型供应商 API Key 和 `agent.studio.trace.persist-full-model-content`。
- 模型供应商普通查询接口只返回密钥是否已配置和掩码，不返回明文。
- 设置接口返回的每一项至少包含 `settingKey`、`settingValue`、`valueType`、`editable`、`description` 和 `source`；`source` 只能是 `SYSTEM_SETTING` 或 `APPLICATION_CONFIG`。
- 普通外部 Agent 详情接口对敏感 header 只返回 `headerName` 和 `secretConfigured` 等元信息，不返回明文，也不返回可回传的掩码占位值。
- `PUT /api/external-agents/{adapterId}` 只更新结构化配置和非敏感字段，不要求前端重传已保存的敏感 secret。
- 敏感 header 的值必须通过 `PUT /api/external-agents/{adapterId}/secrets` 单独更新；未出现在请求中的敏感 secret 保持原值不变。

## 4. 同步运行响应

```json
{
  "runId": "string",
  "agentKey": "string",
  "workflowVersionId": "string",
  "status": "SUCCESS | FAILED | TIMEOUT",
  "output": {},
  "error": {
    "code": "string",
    "message": "中文错误说明"
  },
  "durationMs": 0
}
```

## 5. 错误原则

- 工作流发布失败返回节点级错误。
- Schema 校验失败包含字段路径。
- Java 方法失败包含 methodKey 和错误摘要。
- AGENT_CALL 失败包含目标 Agent 和子运行编号，或无法创建子运行的原因。
- EXTERNAL_AGENT 失败包含适配器、退出码或 HTTP 状态、错误摘要。
- Eval 失败能定位到失败用例和失败断言。

## 6. 运行与验收绑定原则

- 正式运行和调试运行都必须返回实际绑定的 `workflowVersionId`。
- 调试接口如果未指定 `workflowVersionId`，只能选择当前草稿指针指向的已持久化版本。
- 接口层不接受“仅存在于前端内存中的未保存画布”直接运行；必须先保存草稿，再使用新的 `workflowVersionId` 调试。
- EvalRun 必须可独立查询，不能只作为“运行验收接口的返回体”存在。
- 每次 `POST /api/eval-suites/{suiteId}/runs` 都必须同步创建 `AgentRun(runType=EVAL)`；`GET /api/eval-runs/{evalRunId}` 必须返回对应 `runId`。
- `GET /api/runs` 默认面向普通运行页，未显式传入 `runType` 时只返回 `API`、`DEBUG`、`AGENT_CALL`；只有显式传入 `runType=EVAL` 时才返回验收配套运行记录。
- 从 NodeRun 创建 EvalCase 的接口必须把来源链路写入用例对象，不能只做无来源复制。

## 7. 分页筛选

列表接口统一支持：

- page。
- pageSize。
- status。
- keyword。
- startedAtFrom / startedAtTo，适用于运行记录。

具体字段由接口设计文档展开。

## 8. OpenAPI 契约

- 后端 Controller 请求、响应、错误模型必须纳入 OpenAPI。
- 前端通过 OpenAPI 生成 API client 类型或请求响应类型。
- 不允许前端手工复制 Java DTO 字段作为长期契约。
- OpenAPI 文件是公共接口设计和前端类型生成的共同输入。
