# 13-部署安全与质量架构设计 v1

## 1. 部署边界

v1 面向本机或内网单用户部署，可以在信任环境内共享访问，但不提供用户隔离、租户隔离和公网安全边界。

## 2. 部署拓扑

```text
用户浏览器
  -> 前端静态资源服务
  -> Spring Boot API 服务
  -> PostgreSQL
  -> OpenAI-compatible 模型供应商
  -> CLI / HTTP 外部 Agent，可选
```

## 3. 配置项

### 3.1 V1 配置键

| 配置 | 生效来源 | 是否进入 `/api/settings` | 默认值 / 说明 |
|------|----------|-------------------------|---------------|
| AGENT_STUDIO_OPENAI_API_KEY | 启动迁移来源 | 否 | 用于初始化默认 OpenAI-compatible 供应商，迁移后不作为长期运行路由事实源 |
| AGENT_STUDIO_OPENAI_BASE_URL | 启动迁移来源 | 否 | 用于初始化默认供应商 Base URL |
| AGENT_STUDIO_OPENAI_DEFAULT_MODEL | 启动迁移来源 | 否 | 用于初始化默认模型供应项 |
| AGENT_STUDIO_SECRET_KEY | 正式部署必填 | 否 | Base64 编码的 32 字节随机值，用于模型供应商 API Key 可逆加密，不通过接口返回 |
| agent.studio.runtime.default-agent-timeout-seconds | `system_setting` 覆盖启动配置 | 是 | 600，Agent 运行总超时默认值 |
| agent.studio.runtime.default-llm-timeout-seconds | `system_setting` 覆盖启动配置 | 是 | 120，LLM 节点默认超时 |
| agent.studio.runtime.default-java-method-timeout-seconds | `system_setting` 覆盖启动配置 | 是 | 30，JAVA_METHOD 节点默认超时 |
| agent.studio.runtime.default-external-agent-timeout-seconds | `system_setting` 覆盖启动配置 | 是 | 600，EXTERNAL_AGENT 节点默认超时 |
| agent.studio.runtime.default-max-steps | `system_setting` 覆盖启动配置 | 是 | 30，默认最大步数 |
| agent.studio.runtime.default-max-agent-call-depth | `system_setting` 覆盖启动配置 | 是 | 3，默认最大 Agent 调用深度 |
| agent.studio.trace.persist-full-model-content | 仅部署配置 | 否 | true，控制模型提示词和模型输出是否持久保存全文 |

### 3.2 `/api/settings` 白名单

v1 的 `GET /api/settings` 和 `PUT /api/settings` 只承载以下运行限制白名单键：

- `agent.studio.runtime.default-agent-timeout-seconds`
- `agent.studio.runtime.default-llm-timeout-seconds`
- `agent.studio.runtime.default-java-method-timeout-seconds`
- `agent.studio.runtime.default-external-agent-timeout-seconds`
- `agent.studio.runtime.default-max-steps`
- `agent.studio.runtime.default-max-agent-call-depth`

补充原则：

- `system_setting` 只保存上述白名单运行限制键的运行时覆盖值，不镜像全部 Spring 启动配置。
- `AGENT_STUDIO_OPENAI_API_KEY`、`AGENT_STUDIO_OPENAI_BASE_URL` 和 `AGENT_STUDIO_OPENAI_DEFAULT_MODEL` 只作为默认供应商初始化来源，不进入 `/api/settings`。
- `AGENT_STUDIO_SECRET_KEY` 只用于后端密钥加解密，不进入 `/api/settings`、OpenAPI 示例、Trace 或前端状态。
- 正式 Docker 部署缺失 `AGENT_STUDIO_SECRET_KEY` 时必须在 `docker compose config` 阶段失败；应用不得在正式部署中自动生成主密钥，也不得提供固定默认主密钥。
- 模型供应商 API Key 由模型供应商密钥接口维护，普通查询接口只返回配置状态和掩码。
- `agent.studio.runtime.default-timeout-seconds` 不是 v1 正式键，禁止再作为兼容或过渡契约继续扩散。

### 3.3 配置优先级

运行时取值优先级固定如下：

- Agent 运行总超时：`workflow_version.runtime_options_json.timeoutSeconds` > `agent_definition.timeout_seconds`（仅用于生成或归一化 WorkflowVersion 默认值） > `agent.studio.runtime.default-agent-timeout-seconds`
- LLM 节点超时：`node.timeoutSeconds` > `agent.studio.runtime.default-llm-timeout-seconds`
- JAVA_METHOD 节点超时：`node.timeoutSeconds` > `agent.studio.runtime.default-java-method-timeout-seconds`
- EXTERNAL_AGENT 节点超时：`node.timeoutSeconds` > `agent.studio.runtime.default-external-agent-timeout-seconds`
- 最大步数：`workflow_version.runtime_options_json.maxSteps` > `agent_definition.max_steps`（仅用于生成或归一化 WorkflowVersion 默认值） > `agent.studio.runtime.default-max-steps`
- 最大 Agent 调用深度：`workflow_version.runtime_options_json.maxAgentCallDepth` > `agent.studio.runtime.default-max-agent-call-depth`

补充原则：

- 已持久化 WorkflowVersion 的执行时真相只来自 `runtime_options_json`，运行内核不得在执行阶段回头读取 Agent 当前主数据重新解释总超时或最大步数。
- `agent_definition.timeout_seconds` 和 `agent_definition.max_steps` 属于 Agent 级默认值，只用于创建初始草稿、补齐缺省配置或展示默认值，不直接覆盖已持久化版本的运行语义。
- 对于白名单配置键，如果 `system_setting` 中不存在覆盖值，则回退到启动配置（`application.yml` / 环境变量）中的默认值。

## 4. 运行数据保留

- v1 默认持久保存 AgentRun、NodeRun、TraceEvent。
- 模型提示词和模型输出默认保存全文，预留配置项控制开关。
- 不默认自动删除运行数据。
- 如后续需要清理策略，应先确认数据保留周期。

## 5. 安全边界

| 风险点 | 处理 |
|--------|------|
| 公网暴露 | v1 不按公网 SaaS 设计，部署说明标明仅用于本机或内网信任环境 |
| 模型供应商 Key 泄露 | API Key 只写不回显，以可逆密文落库，不写入 Trace 和 OpenAPI 示例 |
| 密钥加密主密钥泄露 | `AGENT_STUDIO_SECRET_KEY` 仅由后端进程读取，不通过接口、日志和前端暴露 |
| 任意 Java 方法调用 | 只允许调用注解注册且启用的方法 |
| 条件表达式注入 | CONDITION 使用声明式规则，不执行脚本 |
| Agent 互调失控 | AGENT_CALL 限制最大调用深度，默认 3，禁止直接调用自己 |
| 外部 Agent 执行风险 | EXTERNAL_AGENT 必须通过适配器注册、默认超时、错误摘要接入 |
| 工具误调用 | TOOL 只能选择已注册且启用的工具，调用前后 Schema 校验 |
| 验收用例误计入 | AI_DRAFT_PENDING EvalCase 不计入正式通过率 |
| Trace 敏感内容 | 默认保存全文，依赖内网单用户部署边界；后续多人共享或公网部署需重做脱敏与访问控制 |

## 6. 测试策略

| 测试类型 | 覆盖对象 |
|----------|----------|
| 领域单元测试 | Agent 状态、WorkflowVersion 发布规则、Schema 锁定规则 |
| 运行内核测试 | 所有 v1 NodeExecutor 和失败路径 |
| Schema 测试 | 输入校验、输出校验、字段路径错误、版本引用 |
| 集成测试 | PostgreSQL、Flyway、MyBatis、运行记录、Trace 写入 |
| Agent 协作测试 | 父子 AgentRun、最大调用深度、子运行失败 |
| 外部 Agent 测试 | CLI/HTTP 适配器、超时、退出码、错误摘要 |
| 节点验收测试 | EvalSuite、EvalCase、EvalRun、断言和关键用例 |
| 前端测试 | 画布保存、节点配置、发布校验、运行详情 |
| 契约测试 | OpenAPI 契约生成、前端类型生成和关键接口兼容性 |

## 7. 编码规范影响

- 后端类、方法和字段必须按项目规范编写中文 Javadoc。
- 注释、日志、错误消息和 UI 文案使用中文。
- 不做双轨 Schema、双轨工作流定义、双轨运行记录。
- 抽象只在真实复用或明显降低复杂度时创建。
