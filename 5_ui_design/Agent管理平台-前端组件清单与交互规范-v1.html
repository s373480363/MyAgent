前端组件清单与交互规范

本文档把前端设计方案进一步下沉到可实现层，明确哪些组件应进入共享层， 哪些组件应归属业务模块，以及每类组件的状态、交互、错误反馈和复用边界。 目标不是罗列控件名，而是建立一套能支撑工作流设计、运行调试、Trace 复盘和节点验收的组件体系。

共享组件优先 业务组件分层 交互规则统一

适用范围

  • React 管理台整体壳层
  • 工作流设计器与节点配置
  • 运行详情、Trace、节点验收

落地原则

  • 共性进入 shared/components
  • 复杂业务组件进入对应 features/*
  • 中文文案、中文错误、明确状态

组件分层

组件分层的核心目标是避免共享层沦为杂物间，也避免业务层重复造轮子。所有组件先判断是否跨模块复用，再决定归属。

壳层组件

承载全局导航、顶部操作区、页面标题、筛选栏、内容容器，不绑定具体业务字段。

AppShell SideNav TopBar PageHeader

共享基础组件

所有业务模块都可能使用，视觉和交互必须统一，不允许同类控件出现多套行为。

Button Tag Card Table Drawer

业务场景组件

依赖特定领域模型，例如 NodeDefinition、AgentRun、SchemaDefinition，不进入共享层。

WorkflowCanvas TraceTimeline EvalRunSummary ModelProviderCatalog ModelOfferingSelector

组件归属判定规则

若一个组件的输入输出完全脱离具体业务对象，且至少会在两个以上模块复用，应进入 shared/components。 若组件内部直接依赖节点类型、运行状态、Schema 版本、验收状态等业务语义，应进入对应 features/*。

组件清单

组件清单按优先级组织。P0 为首批必须稳定落地的组件，P1 为增强组件，P2 为后续优化组件。

组件名 归属建议 优先级 职责 关键状态与交互
AppShell shared/components P0 统一左导航、顶部栏、内容容器、全局反馈入口。 支持导航折叠、当前路由高亮、全局 loading、全局错误提示挂载。
PageHeader shared/components P0 页面标题、副标题、版本状态、右侧主操作区。 支持 breadcrumbs、状态标签、主次按钮、次级说明。
DataTable shared/components P0 承载 Agent、运行记录、Schema、方法、工具、模型供应商、模型供应项、验收集等列表。 支持 loading、空态、错误态、分页、行内操作、列宽控制。
StatusTag shared/components P0 统一展示 Draft、Published、Running、Failed 等状态。 颜色、图标、文案必须固定映射，不允许同状态多种样式。
JsonViewer shared/components P0 展示结构化输入输出、Schema、Trace detail。 支持折叠、复制、搜索、长文本滚动、高亮字段路径。
SchemaFormRenderer shared/forms P0 基于 JSON Schema 渲染调试输入、配置表单和用例输入。 支持必填、枚举、默认值、数组、对象、字段说明与校验提示。
WorkflowCanvas features/workflow P0 封装 React Flow 画布、缩放、拖拽、选中、边连线。 支持未保存状态、错误高亮、运行路径高亮、框选、空白画布操作。
NodePalette features/workflow P0 按节点类型展示 START、LLM、CONDITION、JAVA_METHOD 等。 支持搜索、分组、拖拽添加、禁用态说明。
NodeConfigPanel features/workflow P0 展示当前节点基础配置、运行配置、高级配置。 切换节点时保留未保存提示；字段按分组折叠；错误字段即时定位。
EdgeConfigPanel features/workflow P1 编辑条件边、默认边、结束边说明与规则。 CONDITION 边缺失默认分支时给出显性错误。
ValidationSummaryDrawer features/workflow P0 汇总发布校验和草稿校验结果。 点击错误项直接定位到对应节点或边，并同步打开配置面板。
RunInputPanel features/runs P0 承载调试输入、运行版本选择、运行模式提示。 支持 Schema 表单、原始 JSON 切换、版本标签与运行按钮禁用控制。
RunResultPanel features/runs P0 展示最终输出、错误摘要、耗时、runId。 成功时突出最终结果，失败时突出错误摘要与详情入口。
NodeRunList features/runs P0 展示本次运行经过的节点序列。 支持节点选中、失败节点高亮、按执行时间顺序展示。
TraceTimeline features/runs P0 展示 MODEL_REQUEST、SCHEMA_VALIDATION、NODE_ERROR 等事件。 摘要默认展开，detail 按需查看；支持按事件类型筛选。
SchemaEditor features/schemas P1 用于维护 JSON Schema、版本信息和来源说明。 锁定版本只读；切换版本需显式提示；保存前校验 JSON 合法性。
EvalSuitePanel features/evals P0 展示验收集目标、阈值、状态、绑定节点。 只有 CONFIRMED 状态允许发起正式运行。
EvalCaseEditor features/evals P0 编辑输入、参考答案、断言规则、确认状态。 AI 草稿用例默认显著标识为待确认，不计入正式通过率。
EvalRunSummary features/evals P1 展示通过率、失败数、关键失败用例和历史对比。 关键用例失败时即使通过率达标也需显性提示失败。

全局交互规范

全局交互要保持一致，用户不能在不同页面里重新学习“保存怎么提示、错误怎么定位、抽屉怎么关闭”。

导航与切换

  • 左侧导航点击后立即高亮当前模块。
  • 页面切换时，如果存在未保存内容，先提示再允许离开。
  • 相同路由下的标签页切换不刷新整页，只切换局部内容。

按钮与主操作

  • 每个页面只允许一个主按钮视觉最强，例如“创建 Agent”“发布工作流”“开始调试”。
  • 次级按钮弱化处理，不与主按钮争夺视觉层级。
  • 危险动作必须二次确认，确认文案说明后果。

加载与空态

  • 首次加载用骨架屏，局部刷新优先用局部 loading。
  • 空态必须给出下一步，例如创建、导入、刷新或返回。
  • 长耗时接口如调试运行，应展示进度感而不是静止等待。

焦点与键盘

  • 所有可交互组件必须有明显焦点态。
  • 弹窗、抽屉打开后焦点进入容器内部，关闭时返回触发源。
  • 画布与表单区域允许分别定义快捷键，但不得相互冲突。

工作流交互规范

工作流设计器是前端最复杂区域，交互要优先解决“找节点、配节点、看错误、发版本、跑调试”五个核心问题。

工作流设计器三区交互模型

左：节点库

支持搜索、分组、拖拽、点击添加。节点卡片需说明节点职责，避免用户只看英文类型名。

中：画布区

支持拖拽、缩放、框选、连线、删除、复制、运行路径高亮、错误节点标记。画布空态应提示从左侧拖入 START 节点开始。

右：配置区

分为基础配置、运行配置、高级配置。切换节点时，如果当前面板有未提交编辑，先提示保存或放弃。

新增节点

用户拖入节点后，系统自动生成默认名称和占位配置，并立即选中该节点打开配置面板。画布中心附近优先落点，避免节点落在可视区外。

节点选中

选中态应包含边框高亮、阴影抬升和右侧配置同步切换。多选时右侧面板不展示复杂配置，只显示批量操作或数量说明。

边与条件编辑

点击边后展示边配置。CONDITION 节点的默认分支必须能被直观看见，若缺失默认分支，节点本体、边列表和顶部校验摘要同时提示。

保存与发布

保存草稿只反馈“已保存”与草稿更新时间；发布动作必须先执行校验。校验失败时打开错误汇总抽屉，点击任一错误项直接定位节点或边。

JSON 视图

JSON 视图作为辅助视图，不抢主编辑入口。打开后应支持只读查看、复制、搜索和回到画布，不允许绕开受控配置直接写入危险字段。

节点视觉规则

  • START、END 使用更明确的入口/出口视觉，不与普通节点混同。
  • LLM、REVIEW、SUMMARY 属于 LLM 类节点，可共享一组视觉家族，模型字段必须使用模型供应项选择器并展示供应商、供应项和上游模型名。
  • JAVA_METHOD、TOOL、EXTERNAL_AGENT、AGENT_CALL 要能看出“调用外部能力”的差异。

画布错误规则

  • 错误节点右上角显示危险角标。
  • hover 错误节点时展示简短错误摘要。
  • 严重错误优先在顶部与右侧面板重复提示，避免只在节点上出现一次。

运行与 Trace 交互规范

运行调试页与运行详情页的首要任务是解释“这次执行发生了什么”，因此结果与过程必须同时可见，但主次要清楚。

调试输入区

默认使用 SchemaFormRenderer，支持切换到原始 JSON。顶部始终展示当前运行版本、运行类型和默认超时。

最终结果区

结果优先于日志。成功时突出最终输出，失败时突出错误摘要、错误节点和跳转运行详情入口。

过程追踪区

使用“节点序列 + 节点详情 + Trace 时间线”三段式结构，不把所有事件塞成一栏长日志。

开始运行

点击运行后，按钮进入 loading，输入区转为只读，页面展示 runId 占位与运行中的状态脉冲效果。若是调试草稿运行，必须明确显示是基于草稿固化版本执行。

运行进行中

NodeRun 序列按执行顺序逐步出现，当前节点高亮。若后端仅支持最终返回，也要在前端展示稳定的等待状态，不制造虚假实时日志。

运行成功

顶部状态切换为 SUCCESS，结果区默认展开最终输出，节点序列停留在 END 节点，Trace 中的 RUN_FINISHED 事件自动滚动到可视区。

运行失败

失败节点在节点序列中变红高亮，中间详情区自动聚焦该节点，右侧时间线自动定位最近一个 NODE_ERROR 或 SCHEMA_VALIDATION 事件。

组件 默认展开内容 收起策略 交互要求
NodeRunList 节点名、类型、状态、耗时 不收起，允许纵向滚动 点击节点同步更新中间详情和右侧 Trace 过滤状态。
NodeRunDetail 输入、输出、错误、Schema 校验结果 长 JSON 折叠，重要错误默认展开 失败时优先展示错误摘要，其次展示原始数据。
TraceTimeline 事件摘要、时间、类型 detail 默认收起 支持按事件类型筛选；AGENT_CALL 事件提供子运行跳转。
RunResultPanel 最终输出与 run 基本信息 无 失败时同时显示错误码、中文错误消息和运行详情入口。

Schema 与表单交互规范

Schema 是统一契约，前端不能把 Schema 当成一段原始 JSON 文本随便塞。表单渲染、字段提示、校验反馈必须围绕同一份 SchemaDefinition 工作。

SchemaFormRenderer 规则

  • 对象字段按逻辑分组展示，数组字段支持新增、删除、排序。
  • 字段标签显示中文名称或稳定字段名，必要时补充 description。
  • 必填项、默认值、枚举项、格式约束必须可视化。

SchemaRefPicker 规则

  • 选择器展示 schemaKey、版本号、状态、是否 locked。
  • 被已发布工作流锁定的版本只读展示，不允许原地改结构。
  • 切换 Schema 版本前提示可能影响的映射与字段。

MappingEditor 规则

  • 输入映射和输出映射使用结构化编辑器，不直接暴露脚本输入框。
  • 受控 JSONPath 子集要有提示和示例。
  • 映射目标字段不存在时立即提示，不等到发布才暴露。

SchemaEditor 规则

  • 编辑区与版本信息区分离，避免误把元信息和结构文本混在一起。
  • 保存前先校验 JSON 与 JSON Schema 结构。
  • 锁定版本进入只读态，编辑按钮不可点击并给出说明。

节点验收交互规范

节点验收不是普通列表页，它需要同时表达对象层级、确认状态、运行结果和失败原因，因此建议使用“三段式”结构：Suite、Case、Run。

左侧：EvalSuite 列表

展示名称、绑定节点、状态、通过阈值、最近运行结果。Suite 切换后刷新中间与右侧区域。

中间：EvalCase 编辑

用于编辑输入、参考答案、断言、是否关键用例、确认状态。AI 草稿必须有明显待确认标识。

右侧：EvalRun 结果

展示通过率、失败数、关键失败用例、历史对比曲线和失败原因。

创建用例

支持手工创建与从 NodeRun 导入。来源于历史 NodeRun 的用例默认标记为 AI_DRAFT_PENDING，并展示来源链路信息。

确认用例

只有用户创建或用户确认的用例才能计入正式通过率。确认动作应是显式操作，不要在保存时隐式确认。

发起验收运行

若当前 Suite 不是 CONFIRMED，运行按钮禁用并给出原因。运行结束后，失败用例默认优先展示而不是隐藏在二级标签页里。

反馈与异常规范

反馈规范决定整套系统是否“可控”。同一个问题，页面提示、列表状态、详情信息和操作按钮的反馈不能互相矛盾。

场景 反馈组件 必须展示的信息 交互要求
保存成功 Toast + 页面局部状态 成功文案、更新时间 不要打断用户；工作流页同时清理未保存标识。
保存失败 Inline Alert + Toast 中文错误消息、失败原因 保留当前输入，不清空表单。
发布校验失败 ValidationSummaryDrawer 节点/边定位、错误摘要、字段提示 点击错误项必须直接定位，不允许用户手动找。
运行失败 RunResultPanel + NodeRunDetail runId、错误码、中文错误消息、失败节点 支持进入运行详情继续排查。
Schema 校验失败 Field Error + Detail Panel 字段路径、错误原因、原始摘要 字段错误优先就地提示，同时在详情中保留完整信息。
危险操作确认 ConfirmDialog 操作名称、影响范围、不可逆说明 确认按钮使用危险色，文案明确，不用“确定/取消”空泛描述。

中文错误文案规则

  • 先写问题，再写修正方向。
  • 优先展示后端 error.message，但要补充前端定位信息。
  • 避免只有错误码，没有人类可理解说明。

状态色固定映射

  • Success 统一绿色，Failed 统一红色，Timeout 偏橙色，Running 使用蓝青色。
  • Draft、Published、History 也应有固定颜色，不因页面不同而变化。
  • 状态色只表达状态，不承担品牌装饰。

实现建议

组件文档最终要服务实现，因此这里给出与现有前端架构一致的落地拆分建议。

建议目录拆分

目录 建议内容
src/shared/components Button、StatusTag、PageHeader、DataTable、JsonViewer、Drawer、Dialog、EmptyState、InlineAlert
src/shared/forms SchemaFormRenderer、FieldRenderer、ArrayField、ObjectField、ValidationMessage
src/features/workflow/components WorkflowCanvas、NodePalette、NodeCard、NodeConfigPanel、EdgeConfigPanel、ValidationSummaryDrawer
src/features/runs/components RunInputPanel、RunResultPanel、NodeRunList、NodeRunDetail、TraceTimeline
src/features/evals/components EvalSuitePanel、EvalCaseEditor、EvalRunSummary、EvalHistoryChart

实现顺序建议

  • P0 先做壳层、按钮、标签、表格、JSON 查看器、SchemaFormRenderer。
  • 第二步做工作流设计器核心链路：节点库、画布、配置面板、校验抽屉。
  • 第三步做运行详情、Trace 和节点验收的复杂组件。
  • 图表、历史对比、增强筛选等放到 P1/P2。

收口结论

这套组件体系必须围绕三件事稳定下来：统一状态语义、统一错误反馈、统一编辑与复盘路径。 只要这三点稳定，页面再复杂也不会失控；反之，即使界面再华丽，用户也只会感觉难用。
