Agent 管理平台
前端设计方案

以工作流设计器为核心，以 Trace 复盘和节点验收为两翼，建立一个兼具控制台质感与编辑器效率的桌面端管理台。 视觉上采用“高明度底色 + 深色工作区 + 冷暖高光”的组合，保证信息密度的同时保留画面张力。

设计目标

  • 看起来专业，操作上不沉闷。
  • 把复杂信息组织成可扫描、可定位、可回溯的界面。
  • 让工作流、运行、Trace、验收四个核心场景各自成立。

约束前提

适配本机或内网单用户部署，不做登录、权限和多租户。 页面应默认桌面优先，移动端只保留基础查看能力。

设计原则

所有页面都沿着同一套方法构建：先建立层级，再组织信息，再提供明确反馈，最后控制动效与节奏。

强层级

首页、详情页、设计器、调试页的主次关系必须一眼分明。最重要的操作始终落在视线第一层，不靠猜。

  • 主行动按钮固定在顶部右侧或首屏首位。
  • 状态、版本、风险信息使用显性标签。

强秩序

使用 12 列栅格和统一间距系统，避免管理后台常见的“信息堆满一屏但没重点”的问题。

  • 页面外边距 24-32px。
  • 卡片内边距 20-24px，局部密度可降至 16px。

强反馈

运行中、失败、超时、校验错误、发布成功都必须有明确视觉语义，不能只靠一段文本解释。

  • 按钮、标签、进度、警示统一状态色。
  • 错误信息直接给出字段或节点定位。

强张力

整体采用“浅色信息层 + 深色工作层”的混合策略，既有专业感，也能给工作流画布、Trace、日志提供足够视觉承载。

  • 首页与表单页偏明亮。
  • 工作流画布、运行详情、时间线可适度转深。

视觉系统

页面不使用普通后台模板色。主色冷静、强调色克制，背景与工作区形成层次，避免单一平涂。

字体策略

  • 标题使用偏衬线字体，强化平台的设计感和识别度。
  • 正文使用清晰的无衬线中文字体，保证密集信息下的可读性。
  • 代码、JSON、Trace 使用等宽字体，利于逐行扫描与对比。

层级与阴影

  • 基础卡片使用轻阴影，避免“贴纸感”过强。
  • 弹窗、抽屉、画布容器使用更高层级的阴影与边界。
  • 阴影只承担层次表达，不承担装饰主角。

间距基线

  • 以 8px 为基础节奏，兼容 4 / 12 / 16 / 24 / 32 / 48 的分层间距。
  • 区域之间优先使用留白，不用额外分隔线制造噪音。
  • 密集区域可以缩紧，但页面全局节奏不能被打乱。

布局总览

整体采用桌面端三类工作台：概览型、编辑型、复盘型。共用同一套壳，局部按任务切换分栏结构。

左侧模块库

用于快速进入功能或拖拽节点。内容要求短、清晰、分组明确。

START / END
LLM / REVIEW / SUMMARY
CONDITION / JAVA_METHOD
EXTERNAL_AGENT / TOOL / AGENT_CALL
画布区
使用深色背景与轻网格，突出节点与连线。节点 hover 轻微抬升，选中状态有蓝青色描边和柔和光晕。 运行路径可通过流动渐变边高亮，帮助用户快速理解执行顺序。

右侧配置面板

按“基础配置 / 运行配置 / 高级配置”三层组织，避免一次性暴露过多字段。

基础：名称、说明、输入/输出 Schema
运行：超时、失败策略、模型/方法/适配器
高级：调试采集、提示词、映射

顶栏

固定高度 64px 左右，承载全局搜索、版本状态、保存、发布和调试。因为没有账号体系，右上角不放用户头像，而放系统状态和环境配置入口。

左侧导航

采用窄而清晰的图文导航，一级信息足够明确。导航顺序严格对应用户任务流：概览、Agent、运行、Schema、验收、设置。

工作区

内容区域按任务选择布局，不把所有信息强行塞进统一表格。概览页重信息聚合，设计器重空间，详情页重追踪。

页面设计

每个页面都围绕“当前任务”组织，不做无意义的装饰。页面设计目标是让用户在 3 秒内看清自己能做什么、当前状态是什么、下一步在哪里。

首页概览

首页是控制中心，不是普通 dashboard。首屏放大核心指标、最近运行、失败数量和快捷入口，形成明确的行动入口。

  • 视觉：大号统计、渐变信息卡、快速入口。
  • 交互：一键进入 Agent 列表、最近运行详情、工作流设计器。
  • 目标：让用户在进入平台后马上知道下一步要做什么。

Agent 列表 / 详情

列表页偏密，详情页偏稳。列表要快速筛选、搜索、切换状态；详情页要把基础信息、版本、运行入口与最近记录串起来。

  • 列表使用紧凑表格 + 状态标签 + 行内快捷动作。
  • 详情使用卡片、标签页和侧边辅助信息。
  • 创建与编辑动作保持明确，不隐藏在二级菜单里。

工作流设计器

这是最强视觉中心。采用左节点库、中画布、右配置三栏结构，顶部固定保存、校验、发布、调试、JSON 视图。

  • 画布使用深色背景、细网格、显性节点状态。
  • 选中节点时打开右侧面板，错误时在节点角标和顶部摘要同时提示。
  • 支持草稿、发布版本与历史版本的清晰区分。

运行调试

左侧是 Schema 驱动的输入表单，右侧是结果与日志，底部或侧边显示 Trace 时间线。页面必须持续强调当前运行版本。

  • 输入区支持折叠和自动生成字段说明。
  • 输出区优先展示最终结果，再展开中间过程。
  • 调试态和正式运行态用颜色与标签明确区分。

运行详情 / Trace

这一页的核心是“还原发生了什么”。左边展示节点链路，中间展示选中节点详情，右边展示事件时间线与父子运行关系。

  • 失败节点必须能直接跳转到错误根因。
  • Trace 采用可扫描时间线，不把所有事件堆成纯文本。
  • 模型输入输出、方法调用、外部 Agent 结果支持全文查看。

Schema / 验收 / 设置

Schema 管理强调版本与锁定；节点验收强调用例确认与通过率；系统设置强调默认模型、超时和非敏感配置。

  • Schema 页面适合双栏：列表 + 编辑器。
  • 验收页面适合三层：Suite、Case、Run。
  • 设置页要收敛，不做冗长表单堆叠。

组件规范

组件要统一风格，但不能一刀切。不同场景允许不同信息密度，前提是状态语义一致、操作感一致、层级一致。

组件 设计要求 使用场景
按钮 主按钮使用渐变或强色，次按钮保持克制，危险按钮必须明显区分。 保存、发布、调试、确认、停用等操作。
标签 使用圆角胶囊，状态色必须稳定，不混用含义。版本、运行、验收、节点类型都依赖标签。 草稿 / 发布 / 历史 / 成功 / 失败 / 超时 / 验收状态。
卡片 卡片承担结构分区，不承担过度装饰。卡片之间的边界比阴影更重要。 首页概览、Agent 详情、设置项、说明块。
表格 列表页使用紧凑表格，支持筛选、排序和行内动作。行高控制在适中范围，避免信息稀释。 Agent 列表、运行记录、Schema、验收集列表。
表单 采用分组标签和字段说明，复杂表单默认折叠高级项。Schema 表单要自动生成，不手工堆表格。 Agent 编辑、节点配置、调试输入、系统设置。
抽屉 / 弹窗 用于轻量查看与确认，避免大段复杂编辑在弹窗里完成。高风险动作先解释，再确认。 创建、停用、删除、版本查看、局部编辑。
时间线 Trace 时间线必须可扫描，可折叠，可定位。事件摘要与详情分离，列表先看摘要。 运行详情、调试页、验收运行。
JSON 查看器 使用等宽字体、缩进与折叠。长文本必须可复制、可搜索、可高亮字段路径。 输出结果、Schema、Trace detail、NodeRun 输入输出。
空态 / 错误态 空态给出下一步，不只是“暂无数据”。错误态必须说清楚修什么，而不是只报错码。 首次进入、过滤无结果、发布失败、运行失败。

状态与动效

动效只负责让结构更清楚，不承担娱乐性。所有反馈都应短促、明确、可预测。

运行状态 Pending / Running / Success / Failed / Timeout / Canceled 使用统一色系。Running 采用青蓝脉冲感，Failed 采用更直接的红色提示，不用花哨动画掩盖错误。
版本状态 Draft、Published、History 的视觉差异要稳定。发布后版本以只读样式呈现，草稿状态允许编辑但必须让用户知道这是可变对象。
校验状态 工作流校验、Schema 校验、验收结果都应支持“顶部摘要 + 字段/节点定位”的双重反馈，防止用户只看到一条长错误消息。

动效策略

卡片 hover 只做轻微上浮和阴影变化，持续时间控制在 160-220ms。工作流画布允许节点选中发光、连线高亮、加载骨架和侧栏渐入，但禁止大面积弹跳或夸张缩放。

空态文案

文案必须告诉用户“下一步做什么”。例如：先创建一个 Agent，再进入工作流设计器。

错误文案

文案必须告诉用户“哪里错了、怎么改”。例如：Condition 节点缺少默认分支，请补充后重新发布。

响应式策略

本产品以桌面端为主，响应式的目标不是把所有复杂功能都塞进手机，而是让不同窗口宽度下都能稳定查看和操作。

宽度区间 布局策略 处理方式
1440px 及以上 完整三栏 / 三层布局 左侧导航、主工作区、右侧配置或时间线同时存在。
1280px - 1439px 收紧侧栏，保留核心主操作 降低导航宽度与面板宽度，维持画布或详情主视图。
1024px - 1279px 辅助面板改抽屉 配置、帮助、筛选、时间线可以收起为侧边抽屉，不破坏主任务流。
768px - 1023px 单列为主 列表、详情、表单改为纵向堆叠，保留最关键的操作按钮。
小于 768px 仅保留基础查看能力 不承诺完整工作流编辑体验，只保留少量查看、搜索和状态浏览。

可读性与可访问性

这是内部工具，不代表可以牺牲可读性。高密度界面越要避免模糊、灰化和低对比。

可读性

  • 正文与背景对比度要足够，不用浅灰字配白底。
  • 长文本、JSON、Trace 应支持滚动、复制、搜索和局部展开。
  • 列表、表单、详情页在视觉上保持同一种排版语气。

可访问性

  • 焦点态必须明显，不能只靠鼠标悬停表达可交互。
  • 状态不能只依赖颜色，必要时要加图标和文字标签。
  • 所有错误信息都要用中文，并尽量指向具体字段或节点。
设计结论： 1. 首页要像控制台，工作流页面要像编辑器，运行详情要像审计台。 2. 统一视觉语言，但允许不同页面为各自任务选择不同密度。 3. 把“看起来好看”建立在“看得清、找得到、改得动”之上。
