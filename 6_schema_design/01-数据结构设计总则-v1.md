# 01 数据结构设计总则 v1

## 1. 设计目标

本文件定义平台 v1 的统一数据结构原则，约束后续所有表设计、JSON 结构、版本策略和字段命名方式。

目标不是给某个单表做局部优化，而是保证 Agent、工作流、Schema、运行记录、Trace 和节点验收之间的结构契约一致。

## 2. 数据分层

| 层级 | 数据类型 | 示例 |
|------|----------|------|
| 主数据 | 长期存在、被引用的业务对象 | AgentDefinition、SchemaDefinition、ToolDefinition |
| 版本数据 | 发布后不可变、可追溯 | WorkflowVersion、Schema 版本 |
| 运行数据 | 每次执行产生的实例记录 | AgentRun、NodeRun |
| Trace 数据 | 运行过程中的事件流 | TraceEvent、AgentMessage |
| 验收数据 | 节点质量验证结果 | EvalSuite、EvalCase、EvalRun |
| 配置数据 | 少量全局配置 | SystemSetting |

## 3. 统一约束

- PostgreSQL 是唯一数据库。
- 关系型字段保存标识、状态、版本、时间、外键和筛选字段。
- JSONB 保存工作流节点、边、运行输入输出、Trace 详情、Schema 内容和验收规则。
- 已发布版本禁止原地修改，只能通过新版本演进。
- 被发布工作流引用的 Schema 版本只读。
- v1 不做多租户，不预留 tenant_id。
- v1 不做登录权限，不预留 user_id 作为强制外键。
- 枚举字段优先使用 `varchar` + `CHECK` 约束，避免后续升级困难。
- 所有时间字段使用 `timestamptz`。
- 长文本使用 `text`，结构化内容使用 `jsonb`。

## 4. 命名规则

### 4.1 数据库表

- 表名统一使用小写蛇形命名。
- 一个表一个核心对象，避免复合语义。
- 主表使用单数形式，例如 `agent_definition`、`workflow_version`。

### 4.2 数据库字段

- 主键统一命名为 `id`。
- 业务标识统一命名为 `xxx_key`、`run_no`、`version_no` 等稳定字段。
- 时间字段统一命名为 `created_at`、`updated_at`、`started_at`、`finished_at`。
- 结构化 JSON 列统一命名为 `xxx_json`。

### 4.3 API 与内部对象

- 对外接口字段使用驼峰命名。
- 数据库字段使用蛇形命名。
- 前后端传输层对象不得直接复用数据库结构类。

## 5. 标识与版本

- `id` 是内部主键，允许使用 bigint 自增或 identity。
- `agent_key`、`schema_key`、`run_no` 等是稳定业务键。
- `version_no` 统一从 1 开始整数递增。
- `schema_definition` 例外使用字段名 `version`，因为产品和架构层已将 Schema 版本命名为 `version`；其语义仍是从 1 开始整数递增版本号。
- 同一个业务键的版本号唯一。
- 已发布版本不可删除、不可覆盖、不可修改结构。

## 6. 删除与归档

v1 默认优先使用状态归档，不依赖物理删除。

- 主数据使用 `status` 表示启用、停用、归档等状态。
- 已发布版本不删除，只允许历史保留。
- 运行数据、Trace 数据和验收数据默认永久保留，后续如增加清理能力再单独设计。

## 7. JSON 存储原则

| 类型 | 存储建议 | 说明 |
|------|----------|------|
| 工作流草稿 | `jsonb` | 保存节点、边、布局、配置 |
| 运行上下文 | `jsonb` | 保存输入、节点输出和中间状态 |
| Trace 详情 | `jsonb` | 保存事件全文、参数和错误摘要 |
| Schema 内容 | `jsonb` | 保存完整 JSON Schema |
| 验收规则 | `jsonb` | 保存断言、评分和参考答案 |

## 8. 关系设计原则

- 外键用于表达稳定且高频的对象引用。
- 对于双向引用或版本指针，允许采用应用层校验或延迟外键。
- 运行记录需要保存冗余的业务键，例如 `agent_key`，方便快速查询和排障。
- Trace 数据要优先服务排障和复盘，不要只存最小化摘要。

## 9. 校验原则

- 创建和更新时必须进行结构校验。
- 发布前必须校验引用对象是否存在且状态可用。
- 运行时必须校验输入输出与 Schema 是否一致。
- 节点验收必须校验用例确认状态和断言规则完整性。

## 10. 兼容性原则

- 结构设计要兼容后续增加节点类型、更多模型供应商和更多外部 Agent 适配器。
- 枚举字段预留扩展空间，但不能用模糊字符串代替明确状态。
- JSON 字段内部结构可以扩展，但必须保持已发布版本的字段语义稳定。
