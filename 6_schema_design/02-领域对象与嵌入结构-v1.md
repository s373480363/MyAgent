# 02 领域对象与嵌入结构 v1

## 1. 设计范围

本文件描述平台中最重要的领域对象及其嵌入结构。这里重点说明字段语义、关系和约束，不展开 SQL 细节。

运行追踪表与验收表的物理 DDL 放在后续文档中。

## 2. AgentDefinition

AgentDefinition 是平台的顶层业务对象，一个 Agent 绑定当前草稿工作流和当前发布工作流。

| 字段 | 说明 | 约束 |
|------|------|------|
| id | 内部主键 | 必填 |
| agent_key | 稳定业务标识 | 唯一、不可重复 |
| name | 中文名称 | 必填 |
| description | 描述 | 可空 |
| status | ENABLED / DISABLED | 必填 |
| system_prompt | Agent 默认系统提示词 | 可空 |
| default_model | 默认模型标识 | 可空 |
| temperature | 默认温度 | 可空 |
| timeout_seconds | Agent 默认超时 | 必填 |
| max_steps | 最大步数 | 必填 |
| current_draft_workflow_version_id | 当前草稿版本 | 可空 |
| current_published_workflow_version_id | 当前发布版本 | 可空 |
| created_at | 创建时间 | 必填 |
| updated_at | 更新时间 | 必填 |

约束：

- `agent_key` 全局唯一。
- 当前草稿版本和当前发布版本必须属于同一个 Agent。
- `status=DISABLED` 时不可用于正式调用，但仍可查看和编辑草稿。

## 3. WorkflowVersion

WorkflowVersion 保存草稿、发布和历史版本。每次发布都生成新的不可变版本。

| 字段 | 说明 | 约束 |
|------|------|------|
| id | 内部主键 | 必填 |
| agent_id | 所属 Agent | 必填 |
| version_no | 版本号，从 1 开始递增 | 必填、唯一 |
| status | DRAFT / PUBLISHED / HISTORY | 必填 |
| nodes_json | 节点定义数组 | 必填 |
| edges_json | 边定义数组 | 必填 |
| runtime_options_json | 运行约束配置 | 必填，固定对象结构 |
| referenced_schema_versions_json | 引用的 Schema 版本列表 | 必填，固定对象数组结构 |
| source_workflow_version_id | 复制来源版本 | 可空 |
| published_at | 发布时间 | 可空 |
| created_at | 创建时间 | 必填 |
| updated_at | 更新时间 | 必填 |

约束：

- `(agent_id, version_no)` 唯一。
- `PUBLISHED` 和 `HISTORY` 版本只读。
- `nodes_json` 和 `edges_json` 必须满足节点与边的结构约定。
- `referenced_schema_versions_json` 必须记录所有被直接引用的 Schema 版本。
- `runtime_options_json` 的正式字段集固定为 `timeoutSeconds`、`maxSteps`、`maxAgentCallDepth`。
- `runtime_options_json` 持久化后必须保存完整对象，不允许留空后在执行阶段再回退读取 Agent 当前配置。
- `referenced_schema_versions_json` 的正式结构固定为对象数组，每项为 `{ schemaId, schemaKey, version }`。
- `referenced_schema_versions_json` 由后端根据节点直接引用派生、去重并稳定排序，不允许同时接受字符串列表和对象列表。
- `version_no` 是工作流版本序号，草稿、发布和历史版本共享同一序列。
- 每个 Agent 最多一个可编辑 `DRAFT`，每个 Agent 最多一个当前 `PUBLISHED`，历史版本可有多个。
- 保存草稿时不得原地覆盖当前 `DRAFT` 行，必须创建新的不可变 `DRAFT` 版本并更新当前草稿指针。
- 发布时不得把当前 `DRAFT` 原地转换为 `PUBLISHED`，必须复制当前草稿内容生成新的不可变 `PUBLISHED` 版本。
- `source_workflow_version_id` 记录复制来源，草稿复制、发布复制都应保留可追溯链路。
- 被替换的旧草稿和旧发布版本转为 `HISTORY` 后保持只读，历史调试运行仍可通过 `workflow_version_id` 复盘。
- 调试运行和正式运行都必须绑定具体 `workflow_version_id`，不能依赖可变画布状态。

建议结构：

```json
{
  "runtime_options_json": {
    "timeoutSeconds": 600,
    "maxSteps": 30,
    "maxAgentCallDepth": 3
  },
  "referenced_schema_versions_json": [
    {
      "schemaId": 12,
      "schemaKey": "summary-output",
      "version": 3
    }
  ]
}
```

## 4. NodeDefinition 嵌入结构

NodeDefinition 是 `WorkflowVersion.nodes_json` 中的单个节点对象。

| 字段 | 说明 |
|------|------|
| nodeId | 节点内部标识，在工作流内唯一 |
| type | 节点类型 |
| name | 节点名称 |
| description | 节点说明 |
| inputSchemaRef | 输入 Schema 引用 |
| outputSchemaRef | 输出 Schema 引用 |
| inputMapping | 输入映射 |
| outputMapping | 输出映射 |
| timeoutSeconds | 节点超时 |
| failurePolicy | 失败策略 |
| config | 节点专属配置 |
| ui | 画布显示信息 |

约束：

- `nodeId` 在同一版本内唯一。
- `type` 必须属于平台支持的节点类型集合。
- `inputSchemaRef` 和 `outputSchemaRef` 是否必填由节点类型决定；配置了 Schema 引用时，对应 Schema 版本必须存在。
- `config` 的内部字段按节点类型细化，不允许随意乱填。
- `description` 用于说明节点业务意图，便于画布编辑和 Trace 排障。
- `timeoutSeconds` 为节点级超时覆盖，不配置时使用节点默认值。
- `failurePolicy` 仅能取节点类型支持的策略集合。

### 4.1 SchemaRef 必填规则

| 节点类型 | inputSchemaRef | outputSchemaRef | 说明 |
|----------|----------------|-----------------|------|
| START | 必填 | 不要求 | 定义 Agent 调用输入结构 |
| END | 不要求 | 必填 | 定义 Agent 最终输出结构 |
| LLM | 可选 | 可选 | 配置 outputSchema 时校验模型结构化输出 |
| REVIEW | 可选 | 可选 | 配置 outputSchema 时校验审核输出 |
| SUMMARY | 可选 | 可选 | 配置 outputSchema 时校验总结输出 |
| CONDITION | 可选 | 不要求 | 通常基于上下文字段判断 |
| JAVA_METHOD | 必填 | 必填 | 方法入参和出参必须有 Schema |
| TOOL | 必填 | 必填 | 工具参数和返回值必须有 Schema |
| AGENT_CALL | 不要求 | 不要求 | 目标 Agent 的 START/END Schema 是实际调用契约 |
| EXTERNAL_AGENT | 可选 | 可选 | 配置 outputSchema 时校验外部 Agent 输出 |

规则：

- 必填表示发布校验必须要求该引用存在。
- 可选表示未配置时跳过对应 Schema 校验，但仍必须满足节点自身配置规则。
- 不要求表示该节点不以本地 SchemaRef 作为核心契约，不能强制实现为必填。

### 4.2 inputMapping 语义

`inputMapping` 用于把工作流上下文中的字段取出，组装成节点输入。

推荐形式：

```json
{
  "question": "$.input.question",
  "content": "$.input.content"
}
```

约定：

- Key 表示节点输入字段名。
- Value 表示从工作流上下文读取的 JSONPath。
- 读取结果必须可通过节点输入 Schema 校验。

### 4.3 outputMapping 语义

`outputMapping` 用于把节点输出写回工作流上下文。

推荐形式：

```json
{
  "$.result.summary": "$.summary",
  "$.result.score": "$.score"
}
```

约定：

- Key 表示工作流上下文写入位置。
- Value 表示节点输出中的来源路径。
- Value 的根对象是当前节点输出，`$` 表示整个节点输出；不存在 `$.nodeOutput` 虚拟根。
- 写回前必须确认目标路径合法且不会破坏已存在的强约束字段。

### 4.4 ui 结构

`ui` 仅用于画布展示，不参与运行语义。

| 字段 | 说明 |
|------|------|
| position.x | X 坐标 |
| position.y | Y 坐标 |
| width | 可选宽度 |
| height | 可选高度 |

## 5. EdgeDefinition 嵌入结构

EdgeDefinition 是 `WorkflowVersion.edges_json` 中的单条边对象。

| 字段 | 说明 |
|------|------|
| edgeId | 边标识，在版本内唯一 |
| sourceNodeId | 源节点 |
| targetNodeId | 目标节点 |
| type | NORMAL / CONDITION / DEFAULT / END |
| condition | 受控 JSON 条件对象 |
| isDefault | 是否默认分支 |
| description | 边说明 |

约束：

- 条件边只能连接 CONDITION 节点的条件分支。
- 每个 CONDITION 节点只能有一条默认边。
- `condition` 必须是受控 JSON 对象，不能保存脚本、函数调用或自由文本表达式。
- END 类型边用于表达显式收口，不应承担复杂业务条件。
- `isDefault` 仅在 CONDITION 节点的默认边上为 true。

### 5.1 条件对象结构

CONDITION 边的 `condition` 建议采用以下结构：

```json
{
  "left": "$.order.amount",
  "operator": "GREATER_THAN",
  "right": 100,
  "valueType": "NUMBER"
}
```

支持的操作符：

- `EQUALS`
- `NOT_EQUALS`
- `CONTAINS`
- `NOT_CONTAINS`
- `GREATER_THAN`
- `GREATER_THAN_OR_EQUALS`
- `LESS_THAN`
- `LESS_THAN_OR_EQUALS`
- `IN`
- `NOT_IN`

规则：

- `left` 必须是受控 JSONPath。
- `operator` 必须来自固定枚举。
- `right` 是字面量或字面量数组。
- 复杂判断应由 JAVA_METHOD 节点先产出结构化结果，再由 CONDITION 节点判断。

## 6. SchemaDefinition

SchemaDefinition 是跨层统一结构契约，供表单、工作流、Java 方法、模型输出和验收使用。

| 字段 | 说明 | 约束 |
|------|------|------|
| id | 内部主键 | 必填 |
| schema_key | 稳定业务标识 | 必填、同版本下唯一键的一部分 |
| version | 版本号 | 必填、从 1 开始递增 |
| name | 中文名称 | 必填 |
| description | 描述 | 可空 |
| json_schema | 完整 JSON Schema | 必填 |
| java_type | 可选 Java 类型 | 可空 |
| created_from | 来源 | 必填 |
| status | DRAFT / ACTIVE / ARCHIVED | 必填 |
| locked | 是否被发布工作流引用锁定 | 必填，系统派生 |
| created_at | 创建时间 | 必填 |
| updated_at | 更新时间 | 必填 |

约束：

- `(schema_key, version)` 唯一。
- `status` 只表达生命周期，不表达锁定来源。
- `locked` 由发布工作流引用关系派生，只能由发布流程或引用重算流程维护，不允许人工接口直接改写。
- 只有 `status=DRAFT` 且 `locked=false` 的版本允许修改结构。
- 已被发布工作流引用的版本必须 `locked=true`，并且结构只读。
- `java_type` 仅用于执行边界或展示，不参与跨层契约决策。
- `created_from` 的枚举必须保留完整来源语义，不得合并简写。

## 7. JavaMethodDefinition

JavaMethodDefinition 描述可被工作流调用的平台注册 Java 方法。

| 字段 | 说明 |
|------|------|
| id | 内部主键 |
| method_key | 方法标识 |
| name | 方法名称 |
| description | 方法描述 |
| bean_name | Spring Bean 名称 |
| method_name | 具体方法名 |
| input_schema_id | 入参 Schema 引用 |
| output_schema_id | 出参 Schema 引用 |
| status | ENABLED / DISABLED |
| created_at | 创建时间 |
| updated_at | 更新时间 |

约束：

- 方法必须属于 Spring Bean。
- 只能调用显式注册的方法。
- 入参和出参都必须能够 JSON 序列化。
- 发布前必须校验对应 Schema 存在且可用。

## 8. ToolDefinition

ToolDefinition 描述平台工具目录中的一个工具能力。

| 字段 | 说明 |
|------|------|
| id | 内部主键 |
| tool_key | 工具标识 |
| name | 工具名称 |
| description | 工具描述 |
| input_schema_id | 参数 Schema |
| output_schema_id | 返回 Schema，必填 |
| executor_type | 工具执行器类型 |
| executor_config_json | 执行器配置 |
| status | ENABLED / DISABLED |
| created_at | 创建时间 |
| updated_at | 更新时间 |

约束：

- LLM 节点允许引用的工具必须来自已注册工具目录。
- TOOL 节点参与输出校验和 outputMapping，因此 `output_schema_id` 必填。
- 不返回业务数据的工具必须引用系统内置空对象 Schema，例如 `system.empty_output`，不能让输出契约为空。
- `executor_config_json` 只保存执行器所需的参数，不保存业务流程。

## 9. ExternalAgentDefinition

ExternalAgentDefinition 描述平台外部 Agent 的适配器定义。

| 字段 | 说明 |
|------|------|
| id | 内部主键 |
| adapter_key | 适配器标识 |
| adapter_type | CODEX_CLI / OPENCODE_CLI / CUSTOM_CLI / CUSTOM_HTTP |
| name | 名称 |
| description | 描述 |
| command_json | 命令或请求配置 |
| working_directory | 工作目录 |
| timeout_seconds | 默认超时 |
| capture_stdout | 是否采集 stdout |
| capture_stderr | 是否采集 stderr |
| capture_git_diff | 是否采集 Git diff |
| output_schema_id | 可选输出 Schema |
| status | ENABLED / DISABLED |
| created_at | 创建时间 |
| updated_at | 更新时间 |

约束：

- 节点只能引用已启用的适配器。
- 命令参数必须按数组或结构化配置保存，不允许把整段 shell 字符串当作唯一真相。
- 调试采集能力是可选项，不应影响基础调用语义。

### 9.1 command_json 结构

`command_json` 必须按适配器类型保存结构化配置。

Codex CLI 示例：

```json
{
  "command": "codex",
  "arguments": ["exec", "--json", "--full-auto", "{prompt}"],
  "resultSource": {
    "type": "STDOUT_JSON_LAST_MESSAGE"
  }
}
```

OpenCode CLI 示例：

```json
{
  "command": "opencode",
  "arguments": ["run", "--format", "json", "{prompt}"],
  "resultSource": {
    "type": "STDOUT_JSON"
  }
}
```

Custom CLI 示例：

```json
{
  "command": "my-agent",
  "arguments": ["run", "--input", "{inputJson}", "--prompt", "{prompt}"],
  "resultSource": {
    "type": "STDOUT_JSON"
  },
  "environment": {
    "MY_AGENT_MODE": "local"
  }
}
```

Custom HTTP 示例：

```json
{
  "method": "POST",
  "url": "http://127.0.0.1:8081/agent/run",
  "headers": {
    "Content-Type": "application/json"
  },
  "bodyTemplate": {
    "prompt": "{prompt}",
    "input": "{inputJson}"
  },
  "resultSource": {
    "type": "HTTP_BODY_JSON"
  }
}
```

规则：

- CLI `arguments` 必须是数组。
- 禁止把命令和参数拼成单个 shell 字符串保存。
- HTTP headers 中的敏感值不得在普通详情接口明文回显。
- `{prompt}` 和 `{inputJson}` 是受控占位符，由适配器渲染，不允许执行脚本。

## 10. SystemSetting

SystemSetting 采用 key-value 模式保存少量系统级配置。

| 字段 | 说明 |
|------|------|
| setting_key | 配置键 |
| setting_value | 配置值 |
| value_type | STRING / NUMBER / BOOLEAN / JSON |
| description | 描述 |
| editable | 是否允许页面编辑 |
| updated_at | 更新时间 |

约束：

- `setting_key` 全局唯一。
- `setting_value` 的解析规则由 `value_type` 决定。
- 该表不承载高频业务数据。

## 11. 对象关系

```text
AgentDefinition
  -> currentDraftWorkflowVersionId
  -> currentPublishedWorkflowVersionId

WorkflowVersion
  -> nodes_json
  -> edges_json
  -> referenced_schema_versions_json

NodeDefinition
  -> inputSchemaRef / outputSchemaRef
  -> inputMapping / outputMapping

JavaMethodDefinition
  -> inputSchemaId / outputSchemaId

ToolDefinition
  -> inputSchemaId / outputSchemaId

ExternalAgentDefinition
  -> outputSchemaId
```

## 12. 设计结论

v1 的结构设计核心是三个稳定锚点：

1. Agent 作为业务顶层入口。
2. WorkflowVersion 作为执行版本边界。
3. SchemaDefinition 作为跨层结构契约。

其余对象都应围绕这三个锚点建立可追溯、可校验、可发布的关系。
