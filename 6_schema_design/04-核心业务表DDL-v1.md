# 04 核心业务表 DDL v1

## 1. 说明

本文件给出核心主数据表的 PostgreSQL DDL 设计。这里以可落地实现为目标，使用 `jsonb` 保存结构化配置，使用 `varchar + CHECK` 保存枚举。

说明：

- 主键统一采用 `bigint generated by default as identity`。
- 复杂结构统一使用 `jsonb`。
- 版本和状态字段尽量显式保留。
- 具体字符长度可在迁移时根据实现再微调，但不要改变字段语义。

## 2. agent_definition

```sql
create table agent_definition (
  id bigint generated by default as identity primary key,
  agent_key varchar(128) not null,
  name varchar(200) not null,
  description text not null default '',
  status varchar(20) not null,
  system_prompt text not null default '',
  default_model_offering_key varchar(128),
  temperature numeric(4,2),
  timeout_seconds integer not null default 600,
  max_steps integer not null default 30,
  current_draft_workflow_version_id bigint,
  current_published_workflow_version_id bigint,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint uq_agent_definition_agent_key unique (agent_key),
  constraint ck_agent_definition_status check (status in ('ENABLED', 'DISABLED')),
  constraint ck_agent_definition_timeout_seconds check (timeout_seconds > 0),
  constraint ck_agent_definition_max_steps check (max_steps > 0),
  constraint ck_agent_definition_temperature check (temperature is null or (temperature >= 0 and temperature <= 2))
);
```

建议索引：

- `agent_definition(status)`
- `agent_definition(updated_at desc)`

## 3. workflow_version

```sql
create table workflow_version (
  id bigint generated by default as identity primary key,
  agent_id bigint not null,
  version_no integer not null,
  status varchar(20) not null,
  nodes_json jsonb not null default '[]'::jsonb,
  edges_json jsonb not null default '[]'::jsonb,
  runtime_options_json jsonb not null,
  referenced_schema_versions_json jsonb not null default '[]'::jsonb,
  source_workflow_version_id bigint,
  published_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint uq_workflow_version_agent_version unique (agent_id, version_no),
  constraint ck_workflow_version_status check (status in ('DRAFT', 'PUBLISHED', 'HISTORY'))
);
```

建议索引：

- `workflow_version(agent_id, status, version_no)`
- `workflow_version(agent_id, created_at desc)`
- `workflow_version(source_workflow_version_id)`

建议唯一约束：

```sql
create unique index uq_workflow_version_one_draft
  on workflow_version(agent_id)
  where status = 'DRAFT';

create unique index uq_workflow_version_one_published
  on workflow_version(agent_id)
  where status = 'PUBLISHED';
```

说明：

- `nodes_json` 保存 NodeDefinition 数组。
- `edges_json` 保存 EdgeDefinition 数组。
- `runtime_options_json` 保存完整运行约束快照，正式结构固定为：

```json
{
  "timeoutSeconds": 600,
  "maxSteps": 30,
  "maxAgentCallDepth": 3
}
```

- `runtime_options_json` 是已持久化 WorkflowVersion 的运行时事实来源；执行时不得再回头读取 Agent 当前主数据重算总超时或最大步数。
- `runtime_options_json` 不使用数据库默认空对象；应用层必须在写入 WorkflowVersion 时显式持久化完整对象。
- `referenced_schema_versions_json` 保存引用的 Schema 版本对象数组，正式结构固定为：

```json
[
  {
    "schemaId": 12,
    "schemaKey": "summary-output",
    "version": 3
  }
]
```

- `referenced_schema_versions_json` 由后端派生、去重并稳定排序，不允许同时支持字符串列表和对象列表。
- 每个 Agent 最多一个 `DRAFT`，由 `agent_definition.current_draft_workflow_version_id` 指向。
- 每个 Agent 最多一个 `PUBLISHED`，由 `agent_definition.current_published_workflow_version_id` 指向。
- `version_no` 是工作流定义序号，草稿、发布和历史版本共享同一递增序列；同一 Agent 下不得重复。
- 保存草稿时不允许原地覆盖当前 `DRAFT` 行；必须创建新的 `DRAFT` 行，写入新的 `version_no`，并更新 `agent_definition.current_draft_workflow_version_id`。
- 被替换的旧 `DRAFT` 行转为 `HISTORY`，内容保持不变；历史调试运行仍可通过 `workflow_version_id` 复盘。
- 发布时不允许把当前 `DRAFT` 行原地转换为 `PUBLISHED`；必须复制当前 `DRAFT` 内容创建新的 `PUBLISHED` 行，写入新的 `version_no` 和 `source_workflow_version_id`。
- 发布成功后更新 `agent_definition.current_published_workflow_version_id`；当前草稿指针保持不变，直到下一次保存草稿生成新的 `DRAFT` 行。
- 旧 `PUBLISHED` 只允许状态转为 `HISTORY`，不得修改节点、边、映射、运行配置和 Schema 引用。
- `DRAFT`、`PUBLISHED` 和 `HISTORY` 内容均不可原地覆盖；当前性只通过 Agent 指针和状态表达。

## 5. model_offering

```sql
create table model_offering (
  id bigint generated by default as identity primary key,
  offering_key varchar(128) not null,
  provider_key varchar(128) not null,
  model_key varchar(128) not null,
  display_name varchar(200) not null,
  upstream_model_name varchar(200) not null,
  default_temperature numeric(4,2),
  status varchar(20) not null,
  description text not null default '',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint uq_model_offering_key unique (offering_key),
  constraint uq_model_offering_provider_upstream unique (provider_key, upstream_model_name),
  constraint ck_model_offering_status check (status in ('ENABLED', 'DISABLED')),
  constraint ck_model_offering_temperature check (default_temperature is null or (default_temperature >= 0 and default_temperature <= 2))
);
```

建议索引：

- `model_offering(provider_key, status)`
- `model_offering(model_key, status)`

说明：

- `provider_key` 应引用 `model_provider.provider_key`。
- `model_key` 是跨供应商模型身份字段，不引用独立模型定义表。
- Agent 默认值和 LLM 类节点配置只引用 `offering_key`。
- API Key 不进入该表。

## 6. schema_definition

```sql
create table schema_definition (
  id bigint generated by default as identity primary key,
  schema_key varchar(128) not null,
  version integer not null,
  name varchar(200) not null,
  description text not null default '',
  json_schema jsonb not null,
  java_type varchar(255) not null default '',
  created_from varchar(40) not null,
  status varchar(20) not null,
  locked boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint uq_schema_definition_key_version unique (schema_key, version),
  constraint ck_schema_definition_created_from check (
    created_from in (
      'USER_CREATED',
      'JAVA_METHOD_SCAN',
      'TOOL_DEFINITION',
      'SYSTEM_BUILTIN',
      'AGENT_INPUT',
      'AGENT_OUTPUT'
    )
  ),
  constraint ck_schema_definition_status check (status in ('DRAFT', 'ACTIVE', 'ARCHIVED'))
);
```

建议索引：

- `schema_definition(schema_key, version desc)`
- `schema_definition(created_from, status)`
- `schema_definition(status, locked)`

说明：

- `status` 只表达生命周期，取值为 `DRAFT`、`ACTIVE`、`ARCHIVED`。
- `locked` 由已发布工作流引用关系派生，只能由发布流程或引用重算流程维护。
- 只有 `status=DRAFT` 且 `locked=false` 的版本允许更新结构。
- 工作流发布时，被引用的 Schema 版本应更新为 `status=ACTIVE` 且 `locked=true`。

## 8. tool_definition

```sql
create table tool_definition (
  id bigint generated by default as identity primary key,
  tool_key varchar(128) not null,
  name varchar(200) not null,
  description text not null default '',
  input_schema_id bigint not null,
  output_schema_id bigint not null,
  executor_type varchar(40) not null,
  executor_config_json jsonb not null default '{}'::jsonb,
  status varchar(20) not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint uq_tool_definition_tool_key unique (tool_key),
  constraint ck_tool_definition_status check (status in ('ENABLED', 'DISABLED'))
);
```

建议索引：

- `tool_definition(status)`
- `tool_definition(executor_type)`

说明：

- TOOL 节点执行后需要明确输出契约，因此 `output_schema_id` 必填。
- 不返回业务数据的工具必须引用系统内置空对象 Schema，例如 `system.empty_output`。
- 禁止通过 `output_schema_id = null` 表示“无需校验”，避免运行时出现隐式分支。

### 10.1 command_json 结构

`command_json` 必须保存结构化配置，不允许保存拼接后的 shell 字符串。

Codex CLI：

```json
{
  "command": "codex",
  "arguments": ["exec", "--json", "--full-auto", "{prompt}"],
  "resultSource": {
    "type": "STDOUT_JSON_LAST_MESSAGE"
  }
}
```

OpenCode CLI：

```json
{
  "command": "opencode",
  "arguments": ["run", "--format", "json", "{prompt}"],
  "resultSource": {
    "type": "STDOUT_JSON"
  }
}
```

Custom CLI：

```json
{
  "command": "my-agent",
  "arguments": ["run", "--input", "{inputJson}", "--prompt", "{prompt}"],
  "resultSource": {
    "type": "STDOUT_JSON"
  },
  "environment": {
    "MY_AGENT_MODE": "local"
  }
}
```

Custom HTTP：

```json
{
  "method": "POST",
  "url": "http://127.0.0.1:8081/agent/run",
  "headers": {
    "Content-Type": "application/json"
  },
  "bodyTemplate": {
    "prompt": "{prompt}",
    "input": "{inputJson}"
  },
  "resultSource": {
    "type": "HTTP_BODY_JSON"
  }
}
```

规则：

- CLI `arguments` 必须是数组。
- `{prompt}` 和 `{inputJson}` 是受控占位符，由适配器负责渲染。
- HTTP headers 中的敏感值不得在普通查询接口明文回显。
- 适配器执行时必须使用参数数组或结构化 HTTP 请求，禁止通过字符串拼接执行 shell。

## 11. 约束补充

### 12.1 外键策略

上面 DDL 只展示主结构。实际落地时建议对下列关系补充外键或应用层校验：

- `workflow_version.agent_id -> agent_definition.id`
- `agent_definition.default_model_offering_key -> model_offering.offering_key`
- `model_offering.provider_key -> model_provider.provider_key`
- `java_method_definition.input_schema_id -> schema_definition.id`
- `java_method_definition.output_schema_id -> schema_definition.id`
- `tool_definition.input_schema_id -> schema_definition.id`
- `tool_definition.output_schema_id -> schema_definition.id`
- `external_agent_definition.output_schema_id -> schema_definition.id`

`agent_definition` 与 `workflow_version` 之间存在双向引用指针，若采用物理外键，建议使用延迟约束或迁移脚本分阶段创建。

### 12.2 字段策略

- `description` 默认不为 null，减少空值分支。
- `jsonb` 列默认空对象或空数组，避免运行时判空散落。
- 版本字段从 1 开始递增。
- 状态字段的值必须和设计文档保持一致。

### 12.3 迁移策略

核心表结构一旦进入开发实现，后续变更优先通过新字段扩展，不要通过改语义的方式复用旧字段。
