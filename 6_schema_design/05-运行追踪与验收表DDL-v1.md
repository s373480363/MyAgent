# 05 运行追踪与验收表 DDL v1

## 1. 说明

本文件定义运行记录、Trace、Agent 消息和节点验收相关表结构。

这些表的共同目标是：让每次运行都可以复盘输入、过程、输出和失败点。

## 2. agent_run

```sql
create table agent_run (
  id bigint generated by default as identity primary key,
  run_no varchar(64) not null,
  agent_id bigint not null,
  agent_key varchar(128) not null,
  workflow_version_id bigint not null,
  parent_run_id bigint,
  run_type varchar(20) not null,
  input_json jsonb not null default '{}'::jsonb,
  output_json jsonb,
  status varchar(20) not null,
  error_message text not null default '',
  started_at timestamptz not null default now(),
  finished_at timestamptz,
  duration_ms bigint,
  constraint uq_agent_run_run_no unique (run_no),
  constraint ck_agent_run_run_type check (run_type in ('DEBUG', 'API', 'AGENT_CALL', 'EVAL')),
  constraint ck_agent_run_status check (status in ('PENDING', 'RUNNING', 'SUCCESS', 'FAILED', 'TIMEOUT', 'CANCELED'))
);
```

建议索引：

- `agent_run(agent_id, status, started_at desc)`
- `agent_run(agent_key, run_type, started_at desc)`
- `agent_run(workflow_version_id, started_at desc)`
- `agent_run(parent_run_id)`

说明：

- `id` 是数据库内部主键，不作为 REST 运行标识暴露。
- `run_no` 是对外运行标识，REST 字段名统一为 `runId`。
- `agent_key` 作为冗余字段，方便按业务键快速查找。
- `parent_run_id` 用于 AGENT_CALL 的父子运行关系。

## 3. node_run

```sql
create table node_run (
  id bigint generated by default as identity primary key,
  run_id bigint not null,
  node_id varchar(128) not null,
  node_name varchar(200) not null,
  node_type varchar(40) not null,
  input_json jsonb not null default '{}'::jsonb,
  output_json jsonb,
  schema_validation_result_json jsonb,
  status varchar(20) not null,
  error_message text not null default '',
  started_at timestamptz not null default now(),
  finished_at timestamptz,
  duration_ms bigint,
  constraint ck_node_run_status check (status in ('PENDING', 'RUNNING', 'SUCCESS', 'FAILED', 'TIMEOUT', 'CANCELED'))
);
```

建议索引：

- `node_run(run_id, started_at asc)`
- `node_run(run_id, node_id)`
- `node_run(node_type, status)`

说明：

- `input_json` 和 `output_json` 保存节点执行全文。
- `schema_validation_result_json` 保存字段级校验结果。
- `node_id` 应与工作流版本中的节点定义对应。

## 4. trace_event

```sql
create table trace_event (
  id bigint generated by default as identity primary key,
  run_id bigint,
  node_run_id bigint,
  eval_run_id bigint,
  event_type varchar(40) not null,
  summary varchar(500) not null,
  detail_json jsonb not null default '{}'::jsonb,
  event_time timestamptz not null default now(),
  constraint ck_trace_event_type check (
    event_type in (
      'MODEL_REQUEST',
      'MODEL_RESPONSE',
      'SCHEMA_VALIDATION',
      'CONDITION_DECISION',
      'JAVA_METHOD_CALL',
      'TOOL_CALL',
      'EXTERNAL_AGENT_CALL',
      'AGENT_CALL',
      'EVAL_CASE_RESULT',
      'NODE_ERROR',
      'RUN_FINISHED'
    )
  )
);
```

建议索引：

- `trace_event(run_id, event_time asc)`
- `trace_event(node_run_id, event_time asc)`
- `trace_event(eval_run_id, event_time asc)`
- `trace_event(event_type, event_time desc)`

说明：

- `summary` 必须独立存储，不能依赖前端解析大 JSON。
- `detail_json` 用于保存事件完整上下文。
- `run_id`、`node_run_id`、`eval_run_id` 可按事件类型选择填充。
- 对节点验收事件，v1 必须同时填充 `run_id=agent_run.id` 和 `eval_run_id=eval_run.id`，禁止只保留单边关联。

## 5. agent_message

```sql
create table agent_message (
  id bigint generated by default as identity primary key,
  parent_run_id bigint not null,
  child_run_id bigint not null,
  source_agent_id bigint not null,
  target_agent_id bigint not null,
  input_json jsonb not null default '{}'::jsonb,
  output_json jsonb,
  summary text not null default '',
  created_at timestamptz not null default now(),
  constraint uq_agent_message_child_run unique (child_run_id)
);
```

建议索引：

- `agent_message(parent_run_id, created_at asc)`
- `agent_message(child_run_id)`
- `agent_message(source_agent_id, target_agent_id)`

说明：

- 一条记录应足够表达一次父子 Agent 调用链路。
- `summary` 只存简短摘要，便于运行详情列表展示。

## 6. eval_suite

```sql
create table eval_suite (
  id bigint generated by default as identity primary key,
  agent_id bigint not null,
  workflow_version_id bigint not null,
  node_id varchar(128) not null,
  name varchar(200) not null,
  goal text not null default '',
  judge_model_offering_key varchar(128) not null,
  judge_temperature numeric(4,2) not null default 0,
  pass_threshold numeric(5,2) not null default 0,
  status varchar(20) not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint ck_eval_suite_status check (status in ('DRAFT', 'CONFIRMED', 'ARCHIVED')),
  constraint ck_eval_suite_judge_temperature check (judge_temperature >= 0 and judge_temperature <= 2),
  constraint ck_eval_suite_pass_threshold check (pass_threshold >= 0 and pass_threshold <= 100)
);
```

建议索引：

- `eval_suite(agent_id, workflow_version_id, node_id)`
- `eval_suite(status)`

说明：

- `pass_threshold` 以百分比表达。
- `node_id` 绑定到单个 LLM 类节点。
- `judge_model_offering_key` 是本验收套件统一使用的 judge 模型供应项。
- `judge_temperature` 是本验收套件统一使用的 judge 温度。
- 状态流转为 `DRAFT -> CONFIRMED -> ARCHIVED`。
- `DRAFT` 可编辑，`CONFIRMED` 可执行正式验收，`ARCHIVED` 不再参与新验收运行。

## 7. eval_case

```sql
create table eval_case (
  id bigint generated by default as identity primary key,
  suite_id bigint not null,
  case_no varchar(64) not null,
  title varchar(200) not null,
  input_json jsonb not null,
  reference_sample_json jsonb,
  judge_rule_text text not null default '',
  hard_checks_json jsonb not null default '[]'::jsonb,
  critical boolean not null default false,
  confirm_status varchar(30) not null,
  source_agent_run_id bigint,
  source_node_run_id bigint,
  source_workflow_version_id bigint,
  source_node_id varchar(128),
  description text not null default '',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint uq_eval_case_case_no unique (suite_id, case_no),
  constraint ck_eval_case_confirm_status check (
    confirm_status in ('USER_CREATED', 'USER_CONFIRMED', 'AI_DRAFT_PENDING', 'ARCHIVED')
  )
);
```

建议索引：

- `eval_case(suite_id, confirm_status)`
- `eval_case(suite_id, critical)`
- `eval_case(source_node_run_id)`
- `eval_case(source_workflow_version_id, source_node_id)`

说明：

- 只有 `USER_CONFIRMED` 可以进入正式 EvalRun；`USER_CREATED` 和 `AI_DRAFT_PENDING` 不计入正式通过率。
- `judge_rule_text` 是 LLM 节点验收主判断规则，确认正式用例前必须非空。
- `reference_sample_json` 是参考样例，不是标准答案。
- `hard_checks_json` 是可选硬约束，不承担自然语言质量主判断。
- `critical=true` 的用例失败时应提升验收失败等级。
- 从历史 NodeRun 生成的用例必须写入 `source_agent_run_id`、`source_node_run_id`、`source_workflow_version_id` 和 `source_node_id`。
- 手工创建用例的来源字段可以为空。

## 8. eval_run

```sql
create table eval_run (
  id bigint generated by default as identity primary key,
  run_no varchar(64) not null,
  suite_id bigint not null,
  agent_id bigint not null,
  workflow_version_id bigint not null,
  node_id varchar(128) not null,
  agent_run_id bigint not null,
  status varchar(20) not null,
  total_case_count integer not null default 0,
  passed_case_count integer not null default 0,
  failed_case_count integer not null default 0,
  pass_rate numeric(5,2) not null default 0,
  summary text not null default '',
  error_message text not null default '',
  started_at timestamptz not null default now(),
  finished_at timestamptz,
  duration_ms bigint,
  constraint uq_eval_run_run_no unique (run_no),
  constraint ck_eval_run_status check (status in ('PENDING', 'RUNNING', 'SUCCESS', 'FAILED', 'TIMEOUT', 'CANCELED'))
);
```

建议索引：

- `eval_run(suite_id, started_at desc)`
- `eval_run(agent_id, status, started_at desc)`
- `eval_run(workflow_version_id, node_id, started_at desc)`

说明：

- `agent_run_id` 是必填字段，指向本次验收运行同步创建的 `agent_run.id`，且对应 `agent_run.run_type='EVAL'`。
- `run_no` 是验收运行对外标识，REST 字段名统一为 `evalRunId`。
- `summary` 用于验收结果总览。

## 9. eval_case_result

```sql
create table eval_case_result (
  id bigint generated by default as identity primary key,
  eval_run_id bigint not null,
  eval_case_id bigint not null,
  output_json jsonb,
  hard_check_result_json jsonb not null default '[]'::jsonb,
  judge_result_json jsonb,
  judge_raw_text text,
  judge_model_offering_key varchar(128),
  judge_prompt_version varchar(32),
  passed boolean not null default false,
  error_message text not null default '',
  duration_ms bigint,
  created_at timestamptz not null default now(),
  constraint uq_eval_case_result unique (eval_run_id, eval_case_id)
);
```

建议索引：

- `eval_case_result(eval_run_id)`
- `eval_case_result(eval_case_id)`
- `eval_case_result(passed)`

说明：

- `hard_check_result_json` 保存硬约束检查结果，固定为数组结构，未配置 hardChecks 时为 `[]`。
- `judge_result_json` 保存 judge LLM 结构化判定结果；hardChecks 失败跳过 judge 时为 null。
- `judge_raw_text` 保存 judge 原始输出；hardChecks 失败跳过 judge 时为 null。
- `judge_model_offering_key` 保存本次 judge 使用的模型供应项；hardChecks 失败跳过 judge 时为 null。
- `judge_prompt_version` 保存本次 judge 使用的提示词版本；hardChecks 失败跳过 judge 时为 null。
- 不设置顶层 `score` 列，分数只从 `judge_result_json.score` 读取。

## 10. 关系补充

- `node_run.run_id -> agent_run.id`

## 11. 节点验收结构切换执行策略

当前系统旧数据会删除，本 DDL 是新安装和空库重建后的唯一结构。

实施要求：

- 直接更新当前 Flyway 基线脚本，使空库迁移后得到本文件定义的 eval_suite、eval_case 和 eval_case_result 结构。
- Java migration 中不得再读取或更新 `reference_answer_json`、`assertions_json`、`score_rule_json`。
- 已经应用旧迁移的本地或测试数据库必须清空后重新执行 Flyway 全量迁移。
- 不编写旧 Eval 字段到新 Eval 字段的转换脚本。
- `agent_message.child_run_id -> agent_run.id`
- `eval_suite.agent_id -> agent_definition.id`
- `eval_suite.workflow_version_id -> workflow_version.id`
- `eval_case.suite_id -> eval_suite.id`
- `eval_case.source_agent_run_id -> agent_run.id`
- `eval_case.source_node_run_id -> node_run.id`
- `eval_case.source_workflow_version_id -> workflow_version.id`
- `eval_run.suite_id -> eval_suite.id`
- `eval_run.agent_id -> agent_definition.id`
- `eval_run.workflow_version_id -> workflow_version.id`
- `eval_case_result.eval_run_id -> eval_run.id`
- `eval_case_result.eval_case_id -> eval_case.id`

## 11. 运行与验收的设计结论

运行和验收表的设计重点不是“记录最少字段”，而是让每个失败都能定位到：

1. 哪个 Agent。
2. 哪个版本。
3. 哪个节点。
4. 哪个输入。
5. 哪个校验。
6. 哪个具体错误。
