# 01 接口设计总则 v1

## 1. 设计目标

本文件约束平台 v1 的所有公共接口，确保 REST API、内部服务接口、运行时接口和适配器接口之间没有重复语义和隐式分叉。

## 2. 接口分层

| 层级 | 典型对象 | 职责 | 禁止事项 |
|------|----------|------|----------|
| Web/API | Controller、Request、Response | 接收 HTTP 请求、参数校验、响应包装 | 不直接写复杂业务规则 |
| Application | ApplicationService、Command、Query、Result | 编排用例、事务边界、权限前置校验 | 不直接执行节点、不直连外部 SDK |
| Runtime | Validator、Compiler、Executor、Writer | 编译工作流、执行节点、写 Trace | 不修改草稿、不绕过版本约束 |
| Registry | Registry、Descriptor | 管理可调用目录 | 不保存运行态 |
| Adapter/Gateway | OpenAI、CLI、HTTP、JavaMethod | 适配外部或技术能力 | 不承载业务决策 |

## 3. 命名规则

- `Command`：会修改系统状态的输入对象。
- `Query`：只读查询对象。
- `Result`：服务返回对象。
- `Request`：HTTP 入参对象。
- `Response`：HTTP 出参对象。
- `Dto`：跨层传输对象。
- `Snapshot`：不可变快照。
- `Descriptor`：目录描述对象。
- `Invocation`：一次执行调用。

## 4. 统一约束

- 对外 REST 前缀固定为 `/api`。
- v1 不做登录、角色、权限、多租户。
- 所有错误消息必须为中文。
- 业务失败必须返回稳定的响应结构。
- 前后端类型以 OpenAPI 为唯一来源，禁止手工维护两套影子 DTO。
- 内部接口可以比 REST 更细，但不能和 REST 语义冲突。

## 5. 响应规范

### 5.1 成功响应

```json
{
  "success": true,
  "data": {},
  "error": null
}
```

### 5.2 失败响应

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "WORKFLOW_VALIDATION_FAILED",
    "message": "工作流校验失败，请修正节点配置后重新发布。",
    "details": []
  }
}
```

### 5.3 HTTP 状态约定

- 业务级成功和业务级失败统一使用稳定 JSON 包装。
- 框架级参数解析错误、序列化错误和未捕获异常仍可走标准 HTTP 4xx/5xx。
- 对前端和上层调用方而言，`success` 是主要判断入口。
- 对运行类接口，若运行记录已经创建，则节点失败、Schema 校验失败、模型失败或超时都属于一次完成的运行结果，必须返回 `success=true`，并在 `data.status` 中表达 `FAILED` 或 `TIMEOUT`，同时保留 `data.runId`。
- 只有请求参数非法、Agent 不存在、Agent 停用、无发布版本、无法创建运行记录等前置失败，才返回 `success=false` 且 `data=null`。

## 6. 参数与时间

- 时间统一使用带时区 ISO-8601 字符串，例如 `2026-05-27T10:00:00+08:00`。
- 分页参数统一为 `page` 和 `pageSize`。
- 列表过滤尽量使用明确字段，不用模糊复合字符串代替结构化条件。
- JSON 字段保持与后端模型一致的语义，不做字段重命名漂移。

## 7. 校验与异常

- 请求级校验应尽早返回。
- 领域级校验由 Application 或 Runtime 承担。
- 外部调用失败要保留最小可定位信息和中文摘要。
- 所有 `details` 应尽量包含字段路径、错误原因和可修复建议。

## 8. 版本原则

- v1 所有公开接口均属于当前稳定契约。
- 同一接口的响应结构不得在未升级版本号的情况下改语义。
- 新增字段允许向后兼容，但不得改变现有字段含义。
- 删除字段需谨慎，优先废弃而不是立即移除。
