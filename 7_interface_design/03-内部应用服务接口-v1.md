# 03 内部应用服务接口 v1

## 1. 定位

本文件定义 Application 层的公共 Java 接口。它们是 Controller、导入导出、批处理和运行编排共同依赖的稳定入口。

这些接口负责业务用例编排，不直接执行节点逻辑，也不直接依赖具体数据库实现。

## 2. 通用约定

- 入参统一使用 `Command` 或 `Query`。
- 出参统一使用 `Result`。
- 查询接口只读，不修改状态。
- 修改接口必须控制事务边界。
- 业务异常统一抛出带错误码的中文异常。

## 3. AgentApplicationService

```java
public interface AgentApplicationService {
    AgentDetailResult createAgent(CreateAgentCommand command);
    AgentDetailResult updateAgent(UpdateAgentCommand command);
    void changeAgentStatus(ChangeAgentStatusCommand command);
    PageResult<AgentListItemResult> listAgents(ListAgentsQuery query);
    AgentDetailResult getAgentDetail(GetAgentDetailQuery query);
}
```

职责：

- 创建 Agent 基础信息。
- 维护 Agent 启停状态。
- 返回 Agent 列表和详情。
- Agent 详情必须包含当前草稿版本摘要、当前发布版本摘要和历史版本入口摘要。
- 不直接操作工作流版本内容。

## 4. WorkflowApplicationService

```java
public interface WorkflowApplicationService {
    WorkflowDraftResult getWorkflowDraft(GetWorkflowDraftQuery query);
    WorkflowDraftResult saveWorkflowDraft(SaveWorkflowDraftCommand command);
    WorkflowDraftResult copyWorkflowDraftFromVersion(CopyWorkflowDraftFromVersionCommand command);
    WorkflowValidationResult validateWorkflowDraft(ValidateWorkflowDraftCommand command);
    WorkflowPublishResult publishWorkflowDraft(PublishWorkflowDraftCommand command);
    PageResult<WorkflowVersionListItemResult> listWorkflowVersions(ListWorkflowVersionsQuery query);
    WorkflowVersionResult getWorkflowVersion(GetWorkflowVersionQuery query);
}
```

职责：

- 保存草稿。
- 从已有版本复制生成新草稿。
- 校验草稿。
- 发布生成不可变版本。
- 管理当前草稿和当前发布指针。
- 返回工作流版本列表和版本详情。

规则：

- 发布必须先校验。
- 发布成功后旧发布版本转为历史只读。
- 保存草稿必须创建新的不可变 `DRAFT` 版本并更新当前草稿指针，不允许覆盖旧草稿内容。
- 保存草稿时若已存在当前草稿，必须在同一事务内把旧草稿转为 `HISTORY`，保证同一 Agent 只有一个当前 `DRAFT`。
- 从历史版本或其他已持久化版本复制新草稿时，必须通过显式命令接口创建新的不可变 `DRAFT` 版本，并写入 `sourceWorkflowVersionId`。
- 普通保存草稿接口不承载“复制历史版本”的隐含语义。
- 发布成功后只更新当前发布指针；当前草稿指针保持不变，直到下一次保存草稿生成新的 `DRAFT` 版本。
- 发布时若已存在当前发布版本，必须在同一事务内把旧发布版本转为 `HISTORY`，保证同一 Agent 只有一个当前 `PUBLISHED`。
- 调试运行已经绑定过的草稿版本必须可按 `workflowVersionId` 完整复盘。

## 5. RunApplicationService

```java
public interface RunApplicationService {
    RunResult runPublishedAgent(RunPublishedAgentCommand command);
    RunResult runDebugAgent(RunDebugAgentCommand command);
    PageResult<RunListItemResult> listRuns(ListRunsQuery query);
    RunDetailResult getRunDetail(GetRunDetailQuery query);
}
```

职责：

- 创建运行记录。
- 调度同步执行。
- 返回运行结果和失败信息。
- 汇总 NodeRun、TraceEvent 和父子运行关系。

规则：

- 正式 API 只能调用发布版本。
- 调试运行可调用草稿版本，但必须绑定已经固化的 `workflowVersionId`。
- 运行成功或失败都必须返回 `runId`。
- `RunDetailResult` 是正式冻结的运行详情聚合 DTO，必须与 `GET /api/runs/{runId}` 的响应结构一一对应，不允许 Controller 或前端再拼第二套字段层级。

## 10. EvalApplicationService

```java
public interface EvalApplicationService {
    PageResult<EvalSuiteListItemResult> listEvalSuites(ListEvalSuitesQuery query);
    EvalSuiteDetailResult createEvalSuite(CreateEvalSuiteCommand command);
    EvalSuiteDetailResult updateEvalSuite(UpdateEvalSuiteCommand command);
    EvalSuiteDetailResult getEvalSuite(GetEvalSuiteQuery query);
    void confirmEvalSuite(ConfirmEvalSuiteCommand command);
    void archiveEvalSuite(ArchiveEvalSuiteCommand command);
    EvalRunResult runEvalSuite(RunEvalSuiteCommand command);
    PageResult<EvalRunListItemResult> listEvalRuns(ListEvalRunsQuery query);
    EvalRunDetailResult getEvalRun(GetEvalRunQuery query);
    PageResult<EvalCaseResultItemResult> listEvalRunResults(ListEvalRunResultsQuery query);
    PageResult<EvalRunHistoryItemResult> getEvalRunHistory(GetEvalRunHistoryQuery query);
}
```

职责：

- 管理验收套件与运行。
- 管理验收套件 DRAFT、CONFIRMED、ARCHIVED 状态流转。
- 汇总验收结果。
- 提供通过率、失败原因和运行摘要。
- 提供 EvalRun 历史列表、详情、失败用例明细和历史通过率对比。
- `runEvalSuite` 必须同步创建 `AgentRun(runType=EVAL)`，返回结果中同时暴露 `evalRunId` 和 `runId`。
- `EvalRunDetailResult`、`EvalCaseResultItemResult` 和 `EvalRunHistoryItemResult` 都属于正式冻结 DTO，字段以对外 REST 契约为准，不再由实现阶段自行补齐。

## 11. EvalCaseApplicationService

```java
public interface EvalCaseApplicationService {
    PageResult<EvalCaseItemResult> listEvalCases(ListEvalCasesQuery query);
    EvalCaseDetailResult createEvalCase(CreateEvalCaseCommand command);
    EvalCaseDetailResult updateEvalCase(UpdateEvalCaseCommand command);
    EvalCaseDetailResult getEvalCase(GetEvalCaseQuery query);
    void archiveEvalCase(ArchiveEvalCaseCommand command);
    void confirmEvalCase(ConfirmEvalCaseCommand command);
    EvalCaseDetailResult createEvalCaseFromNodeRun(CreateEvalCaseFromNodeRunCommand command);
}
```

职责：

- 维护验收用例的创建、修改、归档和确认。
- 管理 AI 草稿用例向正式用例的确认流转。
- 支持从历史 NodeRun 生成待确认 EvalCase，并写入来源追踪字段。

说明：

- EvalCase 属于 V1 全量范围，对外 REST 必须暴露创建、编辑、确认、归档和查询接口。
- `USER_CREATED` 和 `AI_DRAFT_PENDING` 用例不可计入正式通过率，只有 `USER_CONFIRMED` 用例可以进入正式 EvalRun。
- `createEvalCaseFromNodeRun` 必须复制 NodeRun 输入为用例输入、复制 NodeRun 输出为参考样例 `referenceSample`，并记录 `sourceAgentRunDbId`、`sourceRunId`、`nodeRunId`、`workflowVersionId`、`nodeId`。其中 `sourceAgentRunDbId` 落库为来源外键，`sourceRunId` 仅用于对外追溯和页面展示。
- EvalCase 确认前必须存在有效 `judgeRule`；`hardChecks` 只作为可选硬约束。
- hardChecks 失败时 EvalCaseResult 直接失败，Application 层不得继续调用 judge LLM。
- EvalCaseResult 不提供顶层 score；分数只来自 `judgeResult.score`。

## 13. 事务与异常

- 创建、保存、发布、更新状态使用短事务。
- 运行相关接口不持有长事务。
- 业务校验失败应返回可读错误，不允许吞掉上下文。
- 对外接口不直接暴露底层数据库异常。

## 14. 内部服务设计结论

Application 层接口的核心目标是把业务用例和技术细节拆开：

1. Controller 只做协议转换。
2. Application 只做用例编排。
3. Runtime 只做执行。
4. Registry 只做目录。
