# 架构单一真相清理 v1 变更目的

## 1. 背景

本次变更来自一次全局架构审计。审计发现当前项目中存在多处旧设计残留、文档与代码不一致、前后端重复维护规则集合的问题。

这些问题的共同本质不是单个 bug，而是系统里出现了多个事实来源。开发人员阅读不同文档、接口或代码时，会得到互相冲突的结论，后续开发容易继续沿着旧路径扩展，形成隐藏技术债务。

用户已经明确以下架构取舍：

- `AGENT_STUDIO_OPENAI_*` 不应该继续存在，系统不需要默认供应商和默认模型。
- 全新安装时允许系统先启动，用户后续在页面配置模型供应商和模型供应项。
- 节点验收产品语义不保留“机器评分规则”和“人工判断规则”两套概念，只保留当前 `scoreRule` 语义。
- `isDefault` 不再作为历史兼容或并行事实保留，需要激进清理。
- 工作流和验收相关枚举追求长期单一真相，由后端只读 metadata 接口提供给前端。
- metadata 接口路径采用 `GET /api/platform-metadata`。
- `agent.studio.trace.persist-full-model-content` 不是正式需求，文档中出现该配置是错误的。
- getter/setter 不需要 Javadoc，注释规范需要按这个口径调整。
- 本次暂时排除 LLM 输出波动性相关问题，不处理审计问题 7。
- 当前系统不需要处理旧数据迁移，所有旧数据都会被删除。
- 多 provider 场景不应继续依赖 Spring Boot starter 自动装配出的静态 OpenAI 客户端；模型调用必须按数据库中的供应商路由动态构造客户端。

## 2. 变更目标

本次变更要把前 6 个审计问题整理为一套清晰、可实施、可验收的修复方案，并作为后续开发的正式依据。

必须达成：

- 移除所有 `AGENT_STUDIO_OPENAI_*` 运行、部署、迁移、文档和测试入口。
- 新安装不自动创建默认模型供应商或默认模型供应项。
- 系统在没有模型供应商时可以启动；模型调用在未配置供应项时失败并返回明确中文错误。
- 节点验收文档统一为当前 `EvalCase` 和 `scoreRule` 合同，删除旧的 `assertionRules`、`judgeRules`、`isCritical`、`confirmationStatus` 等旧表述。
- 工作流默认边只使用 `type=DEFAULT`，删除 `isDefault` 字段、接口契约、前端类型、后端校验、测试和文档残留。
- 新增后端只读 metadata 接口，向前端提供节点类型、模型节点类型、条件操作符、条件值类型、验收断言类型等运行时枚举。
- metadata 接口固定为 `GET /api/platform-metadata`。
- 前端工作流编辑器和验收页面不得继续在页面组件里手工维护上述枚举真相。
- 删除 `agent.studio.trace.persist-full-model-content` 的所有文档表述，不补实现。
- 调整开发规范，明确 getter/setter 不需要 Javadoc。
- 所有相关全局文档与本变更方案保持一致。

## 3. 非目标

- 本次不处理 LLM 输出波动性、参考答案脆弱基线和断言推荐策略。
- 本次不引入默认模型、默认供应商、默认供应项或启动期自动初始化模型目录。
- 本次不保留 `AGENT_STUDIO_OPENAI_*` 到新配置名的迁移桥接。
- 本次不保留 `isDefault` 的读取兼容、保存归一化或历史导入分支。
- 本次不编写旧数据迁移脚本，旧数据由环境清理直接删除。
- 本次不继续使用 Spring Boot OpenAI starter 的自动配置能力作为模型调用入口。
- 本次不实现 `agent.studio.trace.persist-full-model-content`。
- 本次不重做模型供应商管理能力，已有供应商目录仍是模型配置的正式入口。
- 本次不引入权限、租户、用户级 API Key 或供应商自动同步模型列表。

## 4. 成功标准

- 除本变更包和历史验收记录外，`AGENT_STUDIO_OPENAI` 不再命中正式代码、正式文档、测试用例和发布说明。
- 除本变更包和历史验收记录外，`isDefault` 不再命中正式代码、正式文档、测试用例和生成类型。
- 除本变更包和历史验收记录外，`persist-full-model-content` 不再命中正式文档或代码。
- 产品、架构、接口、schema、用户手册中的 EvalCase 字段命名一致。
- 前端枚举选项来自 `GET /api/platform-metadata` 或由该接口的 OpenAPI 类型驱动，不再由页面组件维护独立事实。
- 无模型供应商的全新环境可以启动并访问页面。
- 未配置模型供应项时，发布校验或运行调用返回明确中文错误，不静默选择默认模型。
- 开发规范明确 getter/setter 不需要 Javadoc，避免规范与实践再次分叉。
