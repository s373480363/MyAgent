## 4. 手工 API 联调

### 4.1 可控上游

- 使用同一 Docker 网络中的本地 mock 容器 `agent-studio-mock-openai` 作为 OpenAI-compatible 上游。
- mock 对任意 `POST` 返回 OpenAI-compatible JSON；当 `model=timeout-model` 时主动 sleep 10 秒。

### 4.2 收紧正式超时窗口

执行：

```powershell
Invoke-RestMethod -Method Put -Uri 'http://127.0.0.1:18080/api/settings' -ContentType 'application/json' -Body '{
  "items": [
    {
      "settingKey": "agent.studio.runtime.default-llm-timeout-seconds",
      "settingValue": "2",
      "valueType": "NUMBER"
    }
  ]
}'
```

结果：

- 平台正式 LLM 超时更新为 `2` 秒。
- 供应商连接测试没有引入新的独立 timeout 配置，仍复用正式平台超时契约。

### 4.3 验收数据

- 供应商：`mockopenai`，`providerId=2`
- 供应项：`mockopenai.fast`
- 供应项：`mockopenai.timeout`

### 4.4 重建后 API 成功路径

执行：

```powershell
POST /api/model-providers/2/test
{
  "offeringKey": "mockopenai.fast",
  "prompt": "ping"
}
```

结果：

- HTTP 总耗时约 `116ms`。
- 返回 `success=true`。
- `data.durationMs=30`。
- 返回消息为“模型供应商连接测试成功。”

### 4.5 重建后 API 超时路径

执行：

```powershell
POST /api/model-providers/2/test
{
  "offeringKey": "mockopenai.timeout",
  "prompt": "ping"
}
```

结果：

- HTTP 总耗时约 `2097ms`。
- 返回 `success=false`。
- 错误码为 `RUN_TIMEOUT`。
- 错误消息为中文：“模型供应商连接测试超时，请稍后重试或检查模型供应商网络连接。”
- 响应体中未出现 API Key、Base URL、完整上游鉴权报文或模型原始输出全文。

### 4.6 上游调用证据

mock 容器日志记录到正式请求路径：

```text
METHOD=POST PATH=/v1/v1/chat/completions
BODY={"messages":[{"content":"ping","role":"user"}],"model":"fast-model","stream":false,"temperature":0.0}
METHOD=POST PATH=/v1/v1/chat/completions
BODY={"messages":[{"content":"ping","role":"user"}],"model":"timeout-model","stream":false,"temperature":0.0}
```

说明：

- 成功路径与超时路径都真正打到了上游模型调用层。
- 供应商连接测试走的是正式模型调用基础设施，不是旁路模拟。

## 6. 结论

- 架构师指出的 P1 阻塞项已修复：供应商连接测试现在复用正式 LLM 超时契约，系统会在正式边界内主动返回，不再裸调上游。
