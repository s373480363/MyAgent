## 4. 实际开发顺序

完整 V1 的开发顺序必须按照“先平台骨架，再数据与契约，再运行内核，再页面与验收”的依赖链推进。

### 4.1 顺序总览

1. 工程骨架与公共基线
2. 数据库模型与迁移
3. 领域模型、应用服务接口和 REST/OpenAPI 契约
4. 注册目录与系统设置
5. 工作流版本与发布规则
6. 运行内核与通用执行链路
7. 节点执行器按能力递进实现
8. 前端页面、画布与调试界面
9. 节点验收功能
10. 集成测试、回归测试、部署与收尾

### 4.2 顺序原则

- 先实现所有模块的共同底座，再做各节点专属逻辑。
- 先完成数据持久化与契约，再接入复杂页面。
- 先实现运行链路和 Trace，再补 UI 美化和辅助功能。
- 先保证完整 V1 的语义正确，再优化局部交互。
- 所有接口和页面的实现必须围绕不可变版本、统一 Schema 和统一 Trace 进行，不允许引入第二套定义。

### 6.2 落地顺序

建议按以下顺序建表和实现仓储：

1. `schema_definition`
2. `java_method_definition`
3. `tool_definition`
4. `external_agent_definition`
5. `agent_definition`

10. `trace_event`
11. `agent_message`
12. `eval_suite`
13. `eval_case`

理由如下：

- 结构契约先落地，后续工作流、方法、工具、外部 Agent 才能共享统一输入输出。
- 主数据先落地，运行数据才能建立稳定外键与查询条件。
- 运行数据最后落地，避免前期接口还没冻结就先写临时表结构。

### 6.3 版本和锁定规则

实现时必须严格遵循：

- `WorkflowVersion` 不可原地修改。
- 保存草稿生成新的不可变 `DRAFT` 版本。
- 发布生成新的不可变 `PUBLISHED` 版本。
- 历史版本只读。
- `SchemaDefinition` 被已发布工作流引用后只读。
- `locked` 由系统根据引用关系派生。
- 运行记录必须绑定当时的 `workflow_version.id`。
- `SchemaDefinition` 的 `DRAFT` 不是 `WorkflowVersion` 那种不可变版本：`status=DRAFT && locked=false` 的 Schema 允许原地更新；需要演进正式版本或旧版本已锁定时，才创建新版本。

### 6.4 索引与约束

首轮实现必须补齐以下约束：

- `agent_key`、`schema_key + version`、`method_key`、`tool_key`、`adapter_key` 唯一约束
- `workflow_version(agent_id, status, version_no)` 查询索引
- `uq_workflow_version_one_draft` 部分唯一索引，保证同一 `agent_id` 同时最多一条 `status='DRAFT'`
- `uq_workflow_version_one_published` 部分唯一索引，保证同一 `agent_id` 同时最多一条 `status='PUBLISHED'`
- `agent_run(agent_id, status, started_at)` 查询索引
- `node_run(run_id, node_id)` 查询索引
- `trace_event(run_id, event_time)` 查询索引
- `eval_run(suite_id, status)` 查询索引
- `eval_case(suite_id, confirm_status)` 查询索引

### 6.5 数据迁移原则

- 所有表结构通过 Flyway 迁移。
- 不靠手工建表。
- 不在后续开发中随意改列语义。
- 结构变更先加新字段，再做迁移，避免复用旧字段承载新语义。

### 7.4 工作流历史版本能力

完整 V1 必须把工作流历史版本查看能力落实到接口和页面，而不能只停留在数据模型层。

工作流历史版本能力必须通过以下接口和服务同时闭环，实施时应直接按接口设计落地：

- `GET /api/agents/{agentId}`
- `POST /api/agents/{agentId}/workflow-draft/copy-from-version`
- `GET /api/agents/{agentId}/workflow-versions`
- `GET /api/agents/{agentId}/workflow-versions/{workflowVersionId}`
- `WorkflowApplicationService.listWorkflowVersions(ListWorkflowVersionsQuery)`
- `WorkflowApplicationService.copyWorkflowDraftFromVersion(CopyWorkflowDraftFromVersionCommand)`

必须满足：

- Agent 详情页可查看当前草稿、当前发布版本和历史版本摘要。
- 工作流设计器可只读打开历史版本。
- 从历史版本复制生成新草稿必须使用显式命令接口，创建新的不可变 `DRAFT` 版本，并写入 `sourceWorkflowVersionId`。
- 普通保存草稿接口只承担当前草稿保存，不承担历史版本复制语义。
- 保存草稿、复制新草稿、发布三条链路都必须在同一事务内完成“新版本创建 + 旧当前版本转 `HISTORY` + Agent 指针更新”，不允许只创建新版本而延后修正当前版本状态。
- 调试页和运行详情页必须展示实际绑定的 `workflowVersionId`，并支持回看对应版本定义。
- 历史版本允许查看和复制生成新草稿，不允许原地修改。

## 8. 运行内核与节点执行

### 8.1 运行内核基础组件

运行内核必须先实现以下组件：

- `WorkflowValidator`
- `WorkflowCompiler`
- `WorkflowContext`
- `NodeExecutorRegistry`
- `MappingService`
- `SchemaValidationService`
- `TraceWriter`
- `RuntimeLimitGuard`

### 8.2 运行主流程

运行链路必须严格按照下面顺序实现：

1. 接收同步运行请求
2. 解析运行目标版本
3. 创建 `AgentRun`
4. 初始化 `WorkflowContext`
5. 编译 `WorkflowVersion`
6. 进入节点执行循环
7. 每个节点前后写 `NodeRun` 和 `TraceEvent`
8. 节点输出写回工作流上下文
9. 到达 `END` 节点后生成最终输出
10. 更新运行状态
11. 返回 `runId`、`status`、`output`、`error`、`durationMs`

运行态 ID 语义必须在实现阶段直接冻结，不允许各模块自行再定义第二套标识规则：

- `runId` 是对外运行标识，对应数据库 `agent_run.run_no`，用于 `GET /api/runs/{runId}`、运行详情跳转、日志展示和跨页面运行回溯；不是 `agent_run.id` 自增主键。
- `evalRunId` 是对外验收运行标识，对应数据库 `eval_run.run_no`，用于 `GET /api/eval-runs/{evalRunId}`、`GET /api/eval-runs/{evalRunId}/results` 和验收历史对比；不是 `eval_run.id` 自增主键。
- `nodeRunId` 在对外接口中是特例，`POST /api/node-runs/{nodeRunId}/eval-cases` 必须直接使用数据库主键 `node_run.id`，不再额外定义 `nodeRunNo` 或其他对外编号。
- 运行时内部允许同时持有数据库主键和对外业务编号，但接口入参、接口返回、Trace 关联、前端跳转和测试断言必须统一按上述规则执行。
- `agent_run.id`、`eval_run.id`、`node_run.id` 只作为持久化层和内部关联主键使用；除 `nodeRunId` 这一显式例外外，不对外暴露为 REST 路径标识。

`MappingService` 必须执行统一映射规则：

- 只支持受控子集：`$`、点路径、数组下标。
- 读取失败、数组越界、类型不匹配直接失败。
- 写回只允许落到业务输出子树，不允许覆盖 `$.input`。
- 缺失中间对象自动创建。
- 不自动创建数组。
- 类型冲突直接失败。
- 目标字段允许显式覆盖。
- `null` 允许写回，最终是否通过由 Schema 校验决定。

### 8.3 节点执行器实现顺序

节点执行器不要按页面顺序写，而要按运行依赖顺序实现：

1. `START`
2. `END`
3. `CONDITION`
4. `JAVA_METHOD`
5. `LLM`
6. `TOOL`
7. `AGENT_CALL`
8. `EXTERNAL_AGENT`
9. `REVIEW`
10. `SUMMARY`

这样做的原因：

- `START` 和 `END` 是所有工作流的边界。
- `CONDITION`、`JAVA_METHOD`、`LLM` 形成第一条真正可跑通的主链路。
- `TOOL`、`AGENT_CALL`、`EXTERNAL_AGENT` 依赖注册目录、调用上下文和 Trace。
- `REVIEW`、`SUMMARY` 本质上是 LLM 类节点的变体，适合在基础执行链路稳定后实现。

### 8.4 各节点实现要点

#### 8.4.1 START

- 校验输入是否符合 `inputSchemaRef`
- 将输入写入工作流上下文
- 初始化首个 `NodeRun`

#### 8.4.2 END

- 读取当前上下文中的最终输出
- 根据 `outputSchemaRef` 校验最终结果
- 写入最终 `AgentRun.outputJson`

#### 8.4.3 CONDITION

- 只使用受控条件对象，不执行脚本
- 按固定操作符求值
- 命中显式分支或默认分支
- 求值异常直接失败

#### 8.4.4 JAVA_METHOD

- 读取 `JavaMethodDefinition`
- 根据 `inputMapping` 取值
- 进行输入 Schema 校验
- JSON 与 POJO 转换
- 调用显式注册的 Spring Bean 方法
- 返回后进行输出 Schema 校验
- 写回上下文
- JSON 转 POJO、POJO 转 JSON、序列化失败统一归为 `JAVA_METHOD` 执行失败。
- 只有输入或输出不符合 Schema 时，才归为 Schema 校验失败。

#### 8.4.5 LLM

- 解析系统提示词、用户提示词模板和模型配置
- 按节点配置覆盖模型、温度和超时
- `outputSchemaRef` 不存在时走普通文本输出模式
- `outputSchemaRef` 存在时统一走结构化输出模式，并在返回后执行 Schema 校验
- 调用 OpenAI 官方 API
- 将请求和响应写入 Trace
- 必须区分“模型调用失败”“结构化响应解析失败”“Schema 校验失败”三类错误

#### 8.4.6 TOOL

- 从注册目录读取工具定义
- 校验参数 Schema
- 调用工具执行器
- 校验返回值 Schema
- 写回结果到上下文

#### 8.4.7 AGENT_CALL

- 校验目标 Agent 已发布
- 校验最大调用深度
- 创建子 `AgentRun`
- 同步执行目标 Agent 的当前发布版本
- 将子运行结果写回父上下文
- 记录父子运行关系
- 子运行超时则当前节点和父运行按 `TIMEOUT` 处理；父运行取消或超时时必须尽力级联取消子运行，子运行落 `CANCELED`

#### 8.4.8 EXTERNAL_AGENT

- 读取外部 Agent 适配器定义
- 渲染提示词模板
- 按 CLI 或 HTTP 适配器调用
- 将适配器结果统一映射为 `ExternalAgentResult`
- 优先按 `resultSource` 提取 `outputJson`；提取失败时退化为统一文本摘要
- 收集退出码、HTTP 状态或文本结果
- 可选执行输出 Schema 校验
- 写回上下文
- `stdout`、`stderr` 和 Git diff 是调试字段，不自动等于业务输出。

#### 8.4.9 REVIEW

- 以 LLM 形式实现审核逻辑
- 输出结论、原因、评分和建议
- 支持结构化输出校验

#### 8.4.10 SUMMARY

- 以 LLM 形式实现总结逻辑
- 支持文本总结和结构化总结
- 支持结构化输出校验

### 8.5 运行限制

必须在运行内核中强制实现：

- Agent 运行超时
- 单节点超时
- 最大步数
- 最大 Agent 调用深度

运行状态传播规则必须统一：

- `FAILED`：模型、Schema、方法、工具、外部 Agent、结构化解析等业务失败。
- `TIMEOUT`：当前运行或当前节点自身超时。
- `CANCELED`：被外部中断，或因父运行取消 / 超时被级联取消。
- 父运行超时或取消时，必须尽力中断活跃子运行和外部进程。

### 8.6 状态和错误

运行状态必须统一为：

- `PENDING`
- `RUNNING`
- `SUCCESS`
- `FAILED`
- `TIMEOUT`
- `CANCELED`

错误处理要求：

- 运行记录一旦创建成功，后续失败也必须返回 `runId`
- 节点失败必须写入 `NodeRun`
- 关键事件必须写入 `TraceEvent`
- 错误消息必须中文
- `runId` 必须稳定对应 `agent_run.run_no`，不允许在 Controller、Application、Runtime 或前端层把数据库主键重新包装成另一套对外运行标识。
- `evalRunId` 必须稳定对应 `eval_run.run_no`，验收详情、结果明细和历史对比页面统一使用同一业务编号。
- `nodeRunId` 在“从 NodeRun 生成 EvalCase”接口中必须稳定对应 `node_run.id`，运行详情页如果提供该动作入口，传递给后端的也必须是该数据库主键。
- `FAILED` 只表示业务失败。
- `TIMEOUT` 只表示当前运行或当前节点自身超时。
- `CANCELED` 只表示外部中断或父运行级联取消。
- `FAILED`、`TIMEOUT`、`CANCELED` 不允许混用。
- 运行详情必须能展示失败节点、超时节点、取消原因、子运行状态和实际绑定的 `workflowVersionId`。

Trace 保存策略：

- V1 默认保存全文。
- `NodeRun.inputJson` 和 `NodeRun.outputJson` 保存全文。
- `MODEL_REQUEST` 和 `MODEL_RESPONSE` 保存完整模型请求与响应。
- 工具、Java 方法的完整结构化输入输出保存全文。
- 外部 Agent 的 stdout、stderr、Git diff 只在对应采集开关开启时保存全文。
- 列表页只读取 `summary`，详情页读取 `detailJson`。

## 9. 前端开发顺序

### 9.1 先做的前端公共能力

前端必须先完成：

- 路由框架

- 通用表单页
- JSON 查看器
- 动态表单
- 代码编辑器
- 状态标签
- 错误提示
- 空态与加载态

### 9.2 页面开发顺序

前端页面按下列顺序实现：

1. `Settings`
2. `Schemas`
3. `Methods`
4. `Tools`
5. `External Agents`
6. `Agents` 列表和详情
7. `Workflow` 设计器
8. `Debug` 调试页
9. `Runs` 列表和详情
10. `Evals`

各页面必须覆盖的完整能力：

- `Schemas`：列表、创建、编辑 DRAFT Schema、查看详情、基于旧版本创建新版本。
- `External Agents`：列表、详情、创建、编辑、启停、连接测试。
- `Agents`：列表、详情、启停、当前草稿、当前发布版本、历史版本摘要与跳转入口。
- `Workflow`：草稿保存、发布校验、发布、历史版本只读打开、从历史版本复制生成新草稿、JSON 视图。
- `Debug`：基于已持久化版本发起调试运行，展示实际绑定的 `workflowVersionId`。
- `Runs`：运行列表、运行详情、NodeRun、TraceEvent、父子运行、失败定位、实际绑定版本回看。
- `Evals`：套件列表/创建/更新/确认/归档，用例列表/创建/从 NodeRun 生成/更新/确认/归档，EvalRun 列表/详情/结果明细/历史对比。

### 9.3 工作流设计器实现顺序

工作流设计器必须按以下顺序实现：

1. 画布渲染
2. 节点拖拽与连线
3. 节点选择与属性面板
4. 节点库和节点配置表单
5. 保存草稿
6. 校验工作流
7. 发布工作流
8. JSON 视图
9. 调试入口

### 9.4 调试与运行详情实现顺序

调试与运行详情页面必须按以下顺序实现：

1. 输入表单
2. 发起调试运行
3. 展示运行摘要
4. 展示 NodeRun 列表
5. 展示 NodeRun 详情
6. 展示 TraceEvent 时间线
7. 展示父子 AgentRun 跳转
8. 展示 Schema 校验失败路径
9. 展示实际绑定的 `workflowVersionId` 并支持回看对应版本定义

## 10. 节点验收开发顺序

节点验收必须在运行链路与 Trace 链路稳定后实现。

### 10.1 实现顺序

1. `EvalSuite`
2. `EvalCase`
3. `EvalRun`
4. `EvalCaseResult`
5. 从 `NodeRun` 生成 `EvalCase`
6. `EvalSuite` 确认和归档
7. `EvalRun` 历史对比
8. 节点验收页面联调

### 10.2 验收规则

- 只对确认后的用例计入正式通过率
- AI 生成的草稿用例默认不计入正式验收
- 失败原因必须能定位到具体断言和具体用例
- 验收结果必须能关联到原始 `AgentRun`、`NodeRun` 和 `workflowVersionId`
- 验收回溯链路中的 ID 语义必须固定：对外返回 `sourceRunId` 对应 `agent_run.run_no`，`sourceNodeRunId` 对应 `node_run.id`，`evalRunId` 对应 `eval_run.run_no`；数据库内部仍使用 `eval_case.source_agent_run_id -> agent_run.id` 作为来源外键。
- 不允许为 `NodeRun` 再派生第二套对外编号；从运行详情页进入“生成验收用例”时，页面和接口统一传递 `node_run.id`。
- 执行顺序固定为“outputSchema 校验 -> 确定性断言 -> 可选 LLM 评分”
- LLM 评分只做辅助，不覆盖正式断言结果
- 关键用例失败必须影响整次验收结论

## 11. 测试与质量控制

### 11.1 后端测试顺序

后端测试必须按以下顺序落地：

1. 领域对象单元测试
2. Schema 校验测试
3. 工作流发布校验测试
4. 运行内核测试
5. 节点执行器测试
6. 数据库集成测试
7. OpenAPI 契约测试
8. 异常与错误码测试

### 11.2 前端测试顺序

前端测试必须按以下顺序落地：

1. 页面渲染测试
2. 表单校验测试
3. 画布保存与发布测试
4. 调试运行测试
5. 运行详情测试
6. 契约类型生成测试

### 11.3 质量闸门

以下情况都不能进入后续开发：

- 版本不可变规则不成立
- 运行失败无法返回 `runId`
- `runId`、`nodeRunId`、`evalRunId` 语义不一致，或出现第二套运行态标识规则
- Trace 不能定位节点输入输出
- Schema 校验无法返回字段路径
- 前端不能通过 OpenAPI 生成稳定类型
- 发布后的工作流无法复盘历史执行

## 13. 实施风险

| 风险 | 影响 | 处理方式 |
|------|------|----------|
| 版本不可变规则实现不严 | 运行追溯失真 | 保存、发布、运行三条链路必须共用同一版本模型 |
| JSONPath 映射执行细则未实现前容易分叉 | 运行、前端和测试出现分叉 | 按受控子集、写回限制、失败规则和 Schema 校验顺序实现 |
| Trace 事件过大 | 数据库体积膨胀 | 先按全文保存，后续再由配置控制压缩或清理策略 |
| LLM 输出结构不稳定 | 调试和验收反复失败 | 统一输出 Schema 校验和中文错误摘要 |
| 外部 Agent 不可控 | 失败定位困难 | 统一适配器、超时、退出码和错误摘要 |
| 页面先于接口完成 | 前后端联调成本上升 | 先契约后页面，先接口后交互 |

## 14. 完成判定

V1 只有在同时满足以下条件时才算完成：

1. 全量功能范围已实现。
2. 全量接口已通过 OpenAPI 输出并联调。
3. 全量节点已可执行。
4. 运行、Trace、验收三条链路都能复盘。
5. 测试结果已记录且关键缺陷已关闭。
6. 文档和用户手册已同步更新。
